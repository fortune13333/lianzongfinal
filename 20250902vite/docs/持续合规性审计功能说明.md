# “链踪”V6.0 功能说明：持续合规性审计引擎

## 1. 功能概述：从“事后分析”到“事前治理”的飞跃

欢迎来到“链踪”V6.0！本次更新标志着项目从一个高效的“记录与分析系统”向一个主动的**“智能治理平台”**的决定性进化。

在之前的版本中，AI 的角色是在配置**提交之后**，对其进行分析、总结和评估。这是一种“事后分析”的模式，能帮助我们理解已经发生的变更。

V6.0 引入的**持续合规性审计引擎 (Continuous Compliance Auditing Engine)**，则将这一模式彻底颠覆。它的核心目标是**“事前治理”**：在任何不合规的配置被记录到区块链**之前**，就将其识别并**拦截**。

现在，管理员不再仅仅是历史的审阅者，更是规则的制定者。他们可以定义一系列代表组织安全策略和最佳实践的“合规策略”。随后，“链踪”的AI审计引擎将化身为一个不知疲倦的、7x24小时在线的审计专家，自动审查每一次提交，确保只有符合规范的配置才能被接受。

这标志着“链踪”的核心价值从**“可追溯性”**深化到了**“可治理性”**。

## 2. 核心原理：AI驱动的“守门人”

该功能的核心是一个在后端代理 (`agent.py`) 中实现的、由AI驱动的“守门人”工作流。

### 2.1 后端 (`agent.py`)：中央审计与决策中心

1.  **策略即代码 (Policy as Code)**: 管理员在前端UI上创建的所有合规策略（例如“禁止使用Telnet”），都会被集中、持久地存储在核心数据文件 `chaintrace_data.json` 中。这成为了整个系统的“法律条文”。

2.  **变更拦截**: 当任何用户（包括管理员）提交一个新的配置时，请求会首先到达后端代理的 `/api/blockchains/{device_id}` 接口。此时，代理并**不会**立即创建新的区块。

3.  **双重AI分析工作流**: 代理会启动一个严谨的、两阶段的AI分析流程：
    *   **第一阶段：合规性审计 (Compliance Audit)**
        *   代理会提取所有**当前已启用**的合规策略，连同“旧配置”和用户提交的“新配置”，一起发送给 Gemini AI。
        *   此时，AI收到的指令经过特殊设计，要求它扮演一位“网络合规审计师”，核心任务是：**逐一评估“新配置”是否违反了给定的任何一条策略**。
        *   AI被强制要求返回一个结构化的JSON报告，其中包含一个总体的 `overall_status` ('passed' 或 'failed') 和一个详细的 `results` 数组，列出每一条策略的检查结果。

    *   **第二阶段：标准智能分析 (Standard Smart Analysis)**
        *   **仅在第一阶段完全通过 (`overall_status: 'passed'`) 的情况下**，代理才会继续执行我们熟悉的、旧有的AI智能分析（生成`diff`, `summary`, `analysis`, `security_risks`）。

4.  **“守门人”决策逻辑**:
    *   **如果合规性审计失败**: 代理会**立即拒绝**本次提交，向前端返回一个 `400 Bad Request` 错误，并将AI生成的详细审计报告作为错误信息一并返回。**新的区块绝对不会被创建**，不合规的变更被成功拦截在“门外”。
    *   **如果合规性审计通过**: 代理会继续完成第二阶段的分析，然后将**合规审计报告**和**智能分析报告**一同打包，嵌入新区块数据的 `compliance_report` 字段中。最后，创建并保存这个包含了“合规证书”的新区块。

### 2.2 前端 (UI)：策略定义与结果可视化

1.  **管理员的策略指挥中心**: 在“管理中心”，管理员拥有一个专属的“合规策略”选项卡，可以在这里通过一个直观的模态框，完成对所有策略的增、删、改、查，以及一键“启用/禁用”。

2.  **透明的审计结果**: 对于每一个成功记录的区块，用户都可以在其“区块详情”弹窗中，看到一个全新的**“合规性审计报告”**模块。它会清晰地展示本次变更的总体合规状态，以及每一条被检查策略的详细结果，实现了审计过程的完全透明化。

## 3. 使用方法：三步实现自动化治理

### 第一步：定义规则 (管理员操作)

1.  以 `admin` 用户登录，进入**管理中心**，切换到 **“合规策略”** 选项卡。
2.  点击“添加新策略”，在弹出的窗口中定义您的规则。

**示例：创建一条“禁止使用不安全的Telnet协议”的策略**
*   **策略名称**: `禁止使用 Telnet`
*   **严重性**: `高 (High)`
*   **描述**: `所有设备必须禁用Telnet，只允许通过SSH进行管理访问，以防止凭证被窃听。`
*   **规则 (AI可解释)**: `配置中不得出现任何包含 'transport input telnet' 的命令行。`

3.  保存策略。默认情况下，新策略是**启用**的。

### 第二步：触发自动审计 (用户操作)

1.  任何一位操作员（或管理员）进入一台设备的详情页。
2.  在配置文本框中，尝试添加或修改一行配置，使其包含 `transport input telnet`。
3.  点击“推送到设备并记录”按钮。

### 第三步：查看结果

*   **场景A：不合规的变更被拦截**
    *   用户的提交会**失败**。
    *   前端会弹出一个明确的错误提示，内容为：“添加配置失败：配置违反了公司的合规策略：【禁止使用 Telnet】”。
    *   配置历史列表中**不会出现**新的区块。不安全的变更被成功阻止。

*   **场景B：合规的变更被记录**
    *   用户的提交**成功**，新的区块被创建。
    *   当打开这个新区块的详情时，在“合规性审计报告”模块中，用户会看到：
        *   **总体状态**: <span style="color: #22c55e;">通过</span>
        *   **详细结果**:
            *   ✅ **禁止使用 Telnet**: 通过 - 配置中未发现启用了Telnet传输的线路。
            *   ✅ **(其他所有已启用的策略)**: 通过 - ...

## 4. 核心价值与毕业设计亮点

此功能是您项目中体现**前瞻性、实用性和技术深度**的皇冠上的明珠。

1.  **从被动到主动的治理模式**: 它将系统从一个被动的“历史记录员”，转变为一个主动的“策略执行者”，实现了从“亡羊补牢”到“防患于未然”的质的飞跃。

2.  **将治理制度“代码化”**: 它将抽象的公司安全制度（例如写在文档里的“安全基线要求”）转化为了机器可以自动理解和强制执行的规则，极大地降低了因人为疏忽导致的安全风险。

3.  **不可篡改的“合规证明”**: 每一个成功上链的配置，都自带一份由AI签发的、不可篡改的“合规证书”。这对于满足内部审计或外部监管要求（如等级保护、SOX法案）具有极高的价值。

4.  **复杂的AI工作流编排**: 它展示了一个远超简单“文本生成”的、复杂的、多阶段的AI工作流（审计 -> 决策 -> 分析 -> 记录）。这证明了您不仅能“使用”AI，更能深刻理解业务逻辑，将AI作为核心组件“编排”进一个健壮的系统中，去解决复杂的现实世界问题。
