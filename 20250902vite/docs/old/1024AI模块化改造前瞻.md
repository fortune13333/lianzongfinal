### **核心理念：从“直接调用”到“通过标准接口调用”**

想象一下您家里的电器插头。无论您买的是哪个品牌的台灯或电脑，它们都使用相同的标准插头，可以插进墙上任何一个标准插座里。

*   **墙上的插座**：就是我们“链踪”系统。
*   **电器的插头**：就是我们为所有AI模型定义的**“标准AI驱动接口 (Standard AI Driver Interface)”**。
*   **台灯、电脑**：就是具体的AI模型，如Gemini, DeepSeek, Kimi等。
*   **为电器装上标准插头**：就是我们为每个AI模型编写一个“驱动 (Driver)”或“适配器 (Adapter)”。

现在，“链踪”的后端就像一个**只认美式插头**的美国插座，您想用一个欧式插头的电器，就必须自己动手改造电线（修改`services.py`）。而我们的目标，是把它改造成一个**万国通用插座**，只要您给电器换上对应的标准插头，它就能用。

### **V9.0 AI 架构演进蓝图**

我们将对前后端的AI调用逻辑，进行一次彻底的、遵循“适配器模式”的重构。

#### **第一部分：后端AI引擎的“革命” (您的核心关切点)**

**1. 定义一个“标准AI驱动接口” (The Contract)**

我们首先要在代码中定义一个所有“AI驱动”都必须遵守的“合同”。在Python中，我们可以创建一个抽象基类或一个简单的约定。这个接口将包含两个核心方法，对应后端的两大AI功能：

```python
# (这是一个概念性的代码，非最终实现)

class AIEngineDriver:
    def analyze_changes(self, previous_config: str, new_config: str, change_description: str) -> Dict[str, Any]:
        """负责执行提交分析和回滚分析"""
        raise NotImplementedError

    def audit_compliance(self, policies: List[Dict], previous_config: str, new_config: str) -> Dict[str, Any]:
        """负责执行合规性审计"""
        raise NotImplementedError
```

**2. 创建一个`ai_drivers`文件夹**

在后端代码的根目录，我们将创建一个新的文件夹，名为`ai_drivers`。所有具体的AI模型适配器，都将存放在这里。

**3. 实现具体的“AI驱动”**

*   **`ai_drivers/gemini_driver.py` (我们现有的逻辑)**
    我们会将`services.py`中所有调用Gemini SDK的代码，**完全迁移**到这个新文件中，并封装成符合我们标准接口的函数。

    ```python
    # ai_drivers/gemini_driver.py
    import google.generativeai as genai

    def analyze_changes(previous_config, new_config, change_description):
        # ... 这里是所有调用Gemini进行分析的Prompt和逻辑 ...
        response = model.generate_content(...)
        return response.json()

    def audit_compliance(policies, previous_config, new_config):
        # ... 这里是所有调用Gemini进行审计的Prompt和逻辑 ...
        response = model.generate_content(...)
        return response.json()
    ```

*   **`ai_drivers/deepseek_driver.py` (您未来要添加的)**
    您只需要在这个文件夹里创建一个新文件，然后实现那两个标准函数即可。

    ```python
    # ai_drivers/deepseek_driver.py
    import httpx # 用于发送HTTP请求

    DEEPSEEK_API_KEY = "..."
    DEEPSEEK_API_URL = "https://api.deepseek.com/chat/completions"

    def analyze_changes(previous_config, new_config, change_description):
        # 1. 编写适合DeepSeek的Prompt
        # 2. 构建发送给DeepSeek API的请求体
        # 3. 使用httpx调用API
        # 4. 解析DeepSeek的返回结果
        # 5. 将结果重组成一个包含'diff', 'summary', 'analysis'等键的字典并返回
        return {"summary": "...", "analysis": "...", ...}

    def audit_compliance(policies, previous_config, new_config):
        # ... 类似地，实现审计逻辑 ...
        return {"overall_status": "...", "results": [...]}
    ```

**4. 改造`config.ini`，增加“驱动选择器”**

我们将`config.ini`中的`[gemini]`部分，升级为一个更通用的`[ai_provider]`部分。

```ini
[ai_provider]
# 在这里选择您想使用的AI驱动。
# 值必须与 ai_drivers 文件夹下的文件名（不含.py）匹配。
# 可选项: gemini, deepseek, ...
driver = gemini

# 您可以将不同模型的API密钥都放在这里，驱动程序会自己来取
gemini_api_key = "AIza..."
deepseek_api_key = "sk-..."
```

**5. 改造`services.py`，实现“动态加载”**

这是整个方案的“魔法”所在。`services.py`将**不再直接导入和使用`google.generativeai`**。取而代之的是，它会在启动时执行以下逻辑：

```python
# services.py (概念性代码)
from core import config
import importlib

# 1. 从config.ini读取要使用的驱动名称
driver_name = config.get('ai_provider', 'driver') # -> "gemini"

# 2. 动态地导入对应的驱动模块
try:
    # 这行代码相当于: import ai_drivers.gemini_driver as ai_driver
    ai_driver_module = importlib.import_module(f"ai_drivers.{driver_name}_driver")
except ImportError:
    raise Exception(f"错误：找不到名为 '{driver_name}' 的AI驱动！")

# 3. 在需要AI的地方，调用标准接口
def perform_add_block(...):
    # ...
    # 不再是 genai.GenerativeModel(...)
    # 而是:
    analysis_results = ai_driver_module.analyze_changes(...)
    compliance_report = ai_driver_module.audit_compliance(...)
    # ...
```

#### **后端替换流程（最终简化版）**

在完成了这次架构重构后，未来您想将后端的AI从Gemini换成DeepSeek，您需要做的**所有工作**将是：
1.  在`ai_drivers`文件夹下，新建一个`deepseek_driver.py`文件，并参考我们提供的模板，用DeepSeek的API实现那两个标准函数。
2.  打开`config.ini`文件。
3.  将`[ai_provider]`下的`driver`从`gemini`改为`deepseek`。
4.  重启`agent.py`。

**完成！** 您无需再触碰`services.py`中任何一行复杂的业务逻辑代码。

#### **第二部分：前端AI引擎的统一**

前端的替换虽然已经比较方便，但它也存在一个“双轨制”问题：默认模式是直接调用Gemini SDK，自定义模式是调用URL。我们可以用同样的设计哲学，将其统一。

**1. `src/ai_drivers`文件夹**

在前端源码中，我们也创建一个`ai_drivers`文件夹。

*   **`src/ai_drivers/gemini.driver.ts`**: 将现有直接调用`@google/genai`的逻辑封装在这里。
*   **`src/ai_drivers/customApi.driver.ts`**: 将现有通过`fetch`调用自定义URL的逻辑封装在这里。

**2. 改造`.env`文件**

增加一个选择器变量。

```env
# 选择前端AI驱动。可选项: 'gemini', 'custom'
VITE_AI_DRIVER=gemini

# 仅在 VITE_AI_DRIVER = 'custom' 时生效
VITE_COMMAND_GENERATION_API_URL=http://...
VITE_CONFIG_CHECK_API_URL=http://...
```

**3. 改造`geminiService.ts`**

这个文件将不再包含具体的AI调用逻辑，而是变成一个**“调度中心”**。它会在应用启动时，检查`VITE_AI_DRIVER`的值，然后动态地决定后续所有AI请求应该由哪个“驱动”来处理。

### **总结与建议**

这个V9.0的“适配器”架构，虽然会让我们对“链踪”的底层代码进行一次较大规模的重构，但它能带来无与伦比的长期价值：

1.  **真正的模块化**: 将AI提供方的具体实现细节，与“链踪”的核心业务逻辑完全解耦。
2.  **极简的替换流程**: 未来更换任何AI模型，都变成了“编写一个标准驱动”+“修改一行配置”的简单操作，完美地实现了您的设想。
3.  **清晰的职责分离**: `services.py`只关心“业务规则”，而`drivers`只关心“如何与特定的AI API对话”。
4.  **易于测试**: 我们可以轻松地编写一个`mock_driver.py`，在没有网络连接或不想消耗API费用的情况下，对后端业务逻辑进行单元测试。
