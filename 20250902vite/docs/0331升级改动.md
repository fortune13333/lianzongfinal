### 3. services.py — `perform_auto_audit` 在模拟模式下尝试 SSH

**文件**: `agentv2/services.py`，`perform_auto_audit` 函数
**问题**: 当用户在模拟模式（`SIM_USER`）下操作时，终端断开连接会触发自动审计，函数会立即尝试 SSH 连接真实设备（`get_running_config`），导致超时报错、错误日志刷屏。
**修复**: 在函数入口增加模拟模式检查，检测到后直接 `return`，不进行任何 SSH 操作：
```python
def perform_auto_audit(device_id: str, operator: str):
    if is_simulation_mode():
        logging.info(f"[AutoAudit] Simulation mode active — skipping SSH auto-audit for device {device_id}.")
        return
    ...
```

### 4. api_routes.py — 登录限流器 `_login_attempts` 内存泄漏

**文件**: `agentv2/api_routes.py`，`_check_login_rate_limit` 函数
**问题**: `_login_attempts` 字典以 IP 地址为键记录登录时间戳。每次检查时会清理过期时间戳，但**不会删除空列表对应的 IP 键本身**。长期运行的服务器上，所有曾经访问过登录接口的 IP 都会永久留在字典中，造成持续增长的内存占用。
**修复**: 清理过期时间戳后，若该 IP 的列表为空则删除对应键：
```diff
  attempts = [t for t in _login_attempts[ip] if now - t < _RATE_LIMIT_WINDOW]
- _login_attempts[ip] = attempts
+ if attempts:
+     _login_attempts[ip] = attempts
+ elif ip in _login_attempts:
+     del _login_attempts[ip]
```

### 5. api_routes.py — 脚本库可绕过 AI 策略合规检查（安全漏洞）

**文件**: `agentv2/api_routes.py`，`execute_script` 函数
**问题**: 脚本执行端点只做了 config.ini 硬规则拦截（`check_command_against_rules`），但**没有调用 AI 策略合规检查**（`perform_pre_deployment_check`）。这意味着拥有 `script:execute` 权限的 operator 可以用脚本库绕过所有合规策略（如 NTP 必须配置、禁止 `no ip access-list` 等）直接修改设备配置，而不会触发任何阻断。
**修复**: 在命令拦截检查之后、SSH 推送之前，为每台设备调用 `perform_pre_deployment_check`，与批量部署模板走相同的合规门控：

```python
# 新增：AI 策略合规检查
try:
    from services import perform_pre_deployment_check
    perform_pre_deployment_check(db, device_id, rendered)
except HTTPException as e:
    results.append({..., "status": "error", "output": f"合规检查未通过: {e.detail}"})
    continue
```

现在脚本库与配置模板具有相同的安全防护层级：硬规则拦截 + AI 策略合规检查，两道门缺一不可。

## 二、功能改进（2 项）

### 5. api_routes.py — `/api/search` 添加 `limit` 分页参数

**文件**: `agentv2/api_routes.py`，`search_configs` 函数
**背景**: 全文搜索会遍历数据库中所有区块，在大型部署（设备多、历史版本多）下可能一次返回数千条记录，造成接口响应慢、前端卡顿。
**改进**: 新增 `limit` 查询参数，默认 `100`，最大 `500`：
```python
@router.get("/api/search")
def search_configs(
    q: str,
    device_id: Optional[str] = None,
    limit: int = 100,          # 新增
    ...
) -> List[Dict[str, Any]]:
    limit = min(limit, 500)    # 防止客户端传超大值
    ...
    for block in blocks:
        if len(results) >= limit:
            break              # 达到上限立即停止遍历
        ...
```
前端 `ConfigSearchModal` 现在可以通过 `?limit=N` 控制返回数量，无需修改即可受益（默认值与旧行为基本一致）。

### 6. scheduler.py — 正确区分 `backup` 与 `config_pull` 任务类型

**文件**: `agentv2/scheduler.py`
**背景**: `ScheduledTask` 模型支持两种 `task_type`：`backup`（备份快照）和 `config_pull`（定期拉取配置）。旧代码中两者行为完全相同，且函数名为 `_run_backup_job`，`config_pull` 类型被静默处理为 backup，语义混乱，也无法在区块链历史中区分自动备份与定期巡检。
**改进**:
- 函数重命名为 `_run_scheduled_job`，增加 `task_type` 参数
- `config_pull` 任务写入区块时使用 `changeType='config_pull'`，与手动备份（`update`）在历史记录中可区分
- 注册任务时对未知 `task_type` 记录 `warning` 并跳过，防止静默失效
- 任务调用时传入 `task_type` 参数：

```python
_scheduler.add_job(
    _run_scheduled_job,
    args=[str(task.id), str(task.device_ids), str(task.name), task_type],  # 新增 task_type
    ...
)
```

## 四、数据库迁移（1 个迁移文件）

### 003_add_block_unique_constraint.py

**文件**: `agentv2/alembic/versions/003_add_block_unique_constraint.py`
**内容**: 为 `blocks` 表的 `(device_id, index)` 组合添加唯一索引，防止同一设备出现重复版本号，从数据库层面保护区块链完整性。

```sql
-- upgrade
CREATE UNIQUE INDEX IF NOT EXISTS uq_blocks_device_index
ON blocks (device_id, "index");

-- downgrade
DROP INDEX IF EXISTS uq_blocks_device_index;
```

> **注意**: SQLite 不支持 `ALTER TABLE ADD CONSTRAINT`，故使用 `CREATE UNIQUE INDEX` 实现等效约束。PostgreSQL/MySQL 部署时可改为标准约束语法。
