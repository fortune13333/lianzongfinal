# 原子权限列表修复记录

日期：2026-03-31

---

## 一、问题描述

在管理员界面编辑 operator 用户时，"额外权限"勾选框列表中缺少多个新功能对应的原子权限项，导致无法为 operator 授予这些功能的访问能力。

**现象**：打开用户编辑弹窗 → 选择 operator 角色 → 展开额外权限区域，只能看到 10 个权限项，缺少：

- `script:manage`（管理脚本库）
- `script:execute`（执行脚本）
- `task:manage`（管理定时任务）
- `device:view`（查看设备 / 发现拓扑）

---

## 二、根本原因分析

### 原因 1：权限定义存在两套独立的数据源

系统中原本就存在两个地方各自维护了一份权限列表：

**数据源 A**：`src/utils/permissions.ts` — 用于前端逻辑判断（`hasPermission`、`canViewAdminPanel`）

```typescript
// permissions.ts（功能逻辑层）
export const ATOMIC_PERMISSIONS = {
    DEVICE_CREATE: 'device:create',
    DEVICE_UPDATE: 'device:update',
    // ...
    SCRIPT_MANAGE: 'script:manage',   // 有
    SCRIPT_EXECUTE: 'script:execute', // 有
    TASK_MANAGE: 'task:manage',       // 有
    // device:view 缺失
} as const;
```

**数据源 B**：`src/components/admin/UserEditModal.tsx` 文件内的**本地常量** — 仅用于渲染权限勾选框

```typescript
// UserEditModal.tsx（UI 展示层，独立的本地定义）
const ATOMIC_PERMISSIONS = [
    { id: 'device:create', label: '创建设备' },
    { id: 'device:update', label: '更新设备' },
    { id: 'device:delete', label: '删除设备' },
    { id: 'rollback:execute', label: '执行回滚' },
    { id: 'startup:write', label: '写入启动配置' },
    { id: 'user:manage', label: '管理用户' },
    { id: 'template:manage', label: '管理模板' },
    { id: 'policy:manage', label: '管理策略' },
    { id: 'system:reset', label: '重置数据' },
    { id: 'system:settings', label: '系统设置' },
    // script:manage 缺失
    // script:execute 缺失
    // task:manage 缺失
    // device:view 缺失
];
```

这两个列表**完全独立**，命名相同但互不引用。`UserEditModal.tsx` 使用的是自己内部的那份，不是 `permissions.ts` 里的那份。

### 原因 2：新功能开发时只更新了逻辑层，忽略了 UI 层

以往每次新增功能（脚本管理 v9.0、拓扑发现 v10.0）都在 `permissions.ts` 中添加了新权限常量，也在后端 `auth_deps.py` 的 `require_permission()` 中使用了这些权限，但**没有同步更新 `UserEditModal.tsx` 内的本地列表**。

### 原因 3：`device:view` 在 `permissions.ts` 中也缺失

`routers/topology.py` 的 `POST /api/topology/discover` 端点使用了 `require_permission("device:view")`，但这个权限字符串从未被添加到前端的 `permissions.ts` 常量对象中，导致前端代码无法通过常量引用它。

## 五、为什么不改为从 `permissions.ts` 直接导入？

一个潜在的更彻底的方案是：`UserEditModal.tsx` 直接 `import { ATOMIC_PERMISSIONS } from '../../utils/permissions'`，从常量对象动态生成显示列表，彻底消除两套定义。

但存在以下障碍：

1. `permissions.ts` 中的常量 key 是英文大写（`DEVICE_CREATE`），value 是权限字符串（`device:create`），**没有中文标签**，若直接使用则无法显示可读的中文名称
2. `permissions.ts` 中也**没有分组信息**

因此本次采用**保留独立列表、同步内容**的方案，并在注释中标注"与 permissions.ts 保持一致"的维护要求。如果未来权限条目进一步增多，可考虑在 `permissions.ts` 中将常量结构扩展为 `{ id, label, group }` 格式，作为唯一数据源。

## 八、预防措施建议

此次问题的根源在于**同一业务概念（权限列表）由两个文件各自维护**，任何一方更新都可能导致不一致。为避免未来再次出现同类问题：

1. **新增功能时检查清单**：每次在后端 `require_permission("xxx")` 中使用新权限时，必须同时更新：
   - `src/utils/permissions.ts` — 添加常量
   - `src/components/admin/UserEditModal.tsx` 的 `PERMISSION_LIST` — 添加 UI 项

2. **注释提示**：`UserEditModal.tsx` 文件头已加注释：
   ```typescript
   // 与 src/utils/permissions.ts 中的 ATOMIC_PERMISSIONS 保持一致
   ```

3. **长期方案**：将 `permissions.ts` 中的常量结构扩展为包含 `label` 和 `group` 字段，作为全系统唯一的权限数据源，`UserEditModal.tsx` 直接导入使用，彻底消除维护两份数据的风险。
