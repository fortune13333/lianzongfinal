# ChainTrace v9.0 功能升级改动记录

**升级日期**: 2026-03-30
**版本**: v9.0（在 P0~P3 安全/质量升级基础上继续）
**测试结果**: 47/47 后端测试 ✅ · 21/21 前端测试 ✅ · `npm run build` 构建成功 ✅

---

## 升级背景

在完成 P0~P3 安全与质量升级后，评估了项目的实际运维适用性，发现以下四个关键功能缺口：

1. **脚本管理体验太弱** — 缺乏批量脚本执行和管理入口
2. **没有定时任务 / 自动备份** — 需要手动触发所有配置归档
3. **配置搜索能力缺失** — 无法跨设备历史快速定位配置片段
4. **Jinja2 模板变量支持不足** — 只支持简单字符串替换，不支持条件/循环等完整模板语法

本次升级全部实现上述四点。

### 1.6 新增调度器模块

**新文件**: `agentv2/scheduler.py`

封装 APScheduler BackgroundScheduler，提供以下公开函数：

| 函数 | 说明 |
|------|------|
| `get_scheduler()` | 返回全局 scheduler 实例 |
| `add_task_job(task)` | 将 ScheduledTask ORM 对象注册为 APScheduler job（先移除同 ID 旧 job） |
| `remove_task_job(task_id)` | 移除指定 task 的 job |
| `init_scheduler(db)` | 从数据库加载所有 `is_enabled=True` 的任务并启动调度器 |

**任务执行逻辑** (`_run_backup_job`)：
1. 从 DB 查询目标设备列表（`["all"]` 则取所有设备）
2. 循环调用 `get_running_config()` 拉取实时配置
3. 调用 `perform_add_block()` 写入区块链（operator 标记为 `scheduler:<任务名>`）
4. 统计成功/失败，调用 `update_task_run_status()` 更新状态
5. 写入审计日志
6. 支持模拟模式（跳过 SSH，仅记录日志）
7. 所有导入均在函数内部，避免模块级循环导入

#### Jinja2 模板渲染（`bulk_deploy_template` 端点）

**改动前**（简单字符串替换）：
```python
rendered_config = template.content.replace("{{ device.name }}", device.name)
rendered_config = rendered_config.replace("{{ device.id }}", device.id)
rendered_config = rendered_config.replace("{{ device.ipAddress }}", device.ipAddress)
```

**改动后**（完整 Jinja2 引擎）：
```python
_tmpl = _jinja_env.from_string(str(template.content))
rendered_config = _tmpl.render(device={
    "name": device.name,
    "id": device.id,
    "ipAddress": device.ipAddress,
    "type": device.type,
})
```

支持变量、条件语句 `{% if %}`、循环 `{% for %}`、注释 `{# #}` 等完整 Jinja2 语法。

#### 新增端点：配置搜索

```
GET /api/search?q=TEXT[&device_id=DEVICE_ID]
```

- 需要 JWT 认证（任意已登录用户均可调用）
- 遍历所有（或指定设备的）Block，解析 `data.config` 字段
- 大小写不敏感匹配，返回匹配行（最多 10 行/block）及版本、时间戳、区块哈希

返回格式示例：
```json
[
  {
    "device_id": "RTR01",
    "block_index": 3,
    "timestamp": "2026-03-30T14:00:00Z",
    "hash": "abc123...",
    "version": 4,
    "matched_lines": ["  ip ospf 1 area 0", "router ospf 1"]
  }
]
```

#### 新增端点：脚本 CRUD

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/api/scripts` | 已登录 | 列出所有脚本 |
| POST | `/api/scripts` | `script:manage` | 创建脚本（重名返回 409） |
| PUT | `/api/scripts/{id}` | `script:manage` | 更新脚本 |
| DELETE | `/api/scripts/{id}` | `script:manage` | 删除脚本 |
| POST | `/api/scripts/{id}/execute` | `script:execute` | 批量执行脚本 |

**脚本执行流程**（`/execute` 端点）：
1. 查询脚本，遍历 `device_ids`
2. Jinja2 渲染（注入 `device.name/id/ipAddress/type`）
3. 命令拦截检查（`check_command_against_rules()`）
4. 模拟模式：返回渲染结果，不实际 SSH
5. 真实模式：ConnectHandler → `send_config_set()`
6. 按设备收集结果，写入审计日志

```bash
pip install jinja2 apscheduler
```

已有 `chaintrace.db` 的环境，服务器启动时会自动执行 `alembic upgrade head` 创建 `scripts` 和 `scheduled_tasks` 两张新表，无需手动操作，也无需删除旧数据库。

### 定时任务时区

APScheduler 调度器以 **UTC** 时区运行。Cron 表达式中的时间为 UTC 时间。例如北京时间（UTC+8）凌晨 2 点对应 Cron：`0 18 * * *`（前一天 UTC 18:00）。

### Jinja2 模板向后兼容

旧模板中使用 `{{ device.name }}` 等语法完全兼容新的 Jinja2 引擎，无需修改。
