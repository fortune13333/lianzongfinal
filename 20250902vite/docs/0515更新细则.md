# 0515 更新细则 — 等保合规增强 Phase 1A：告警通知系统

## 概述

本次更新实现了等保合规三期计划的第一阶段——告警通知系统。该系统是后续所有等保功能（安全事件检测、设备状态监控增强）的基础设施，提供了多渠道告警投递能力和完整的规则管理界面。

---

## 一、新建文件

### 1.1 `agentv2/notifications.py` — 通知投递服务

三个投递通道，使用 Python 标准库实现，未引入新依赖：

| 函数 | 通道 | 实现方式 |
|------|------|----------|
| `send_email_notification(config, title, message)` | 邮件 (SMTP) | `smtplib.SMTP_SSL`，TLS 加密，支持多收件人 |
| `send_wechat_work_notification(config, title, message)` | 企业微信 | `urllib` POST webhook，Markdown 格式 |
| `send_dingtalk_notification(config, title, message)` | 钉钉 | `urllib` POST webhook，支持 HMAC-SHA256 签名 |

核心调度函数 `deliver_notification(rule, title, message, db)`：
- 根据规则的 `channel` 字段派发到对应通道
- 调用 `crud.create_alert()` 创建告警记录
- 返回布尔值表示是否投递成功

### 1.2 `agentv2/alembic/versions/005_add_notifications.py` — 数据库迁移

- **Revision ID**: 005
- **Down Revision**: 004
- 使用 `IF NOT EXISTS` 语义，兼容已有数据库
- 创建 `notification_rules` 表：id, name, event_type, channel, channel_config (JSON Text), is_enabled, created_by, created_at, updated_at
- 创建 `alerts` 表：id (自增), rule_id (FK → notification_rules.id, ON DELETE SET NULL), event_type, title, message, severity, source, is_sent, sent_at, created_at

### 1.3 `src/store/slices/alertSlice.ts` — 前端告警状态管理

Zustand slice，提供 4 个异步 action：
- `createNotificationRule(rule)` — 创建告警规则
- `updateNotificationRule(ruleId, rule)` — 更新告警规则
- `deleteNotificationRule(ruleId)` — 删除告警规则
- `testNotificationRule(ruleId)` — 测试发送通知

所有操作通过 `apiFetch` 走 JWT 认证，完成后自动调用 `fetchData(true)` 刷新数据。

### 1.4 `src/components/admin/AlertRuleModal.tsx` — 告警规则编辑弹窗

- 支持创建/编辑两种模式
- 动态表单：选择通道后显示对应配置项
  - 邮件通道：SMTP 服务器、端口、发件邮箱、密码/授权码、收件邮箱列表
  - 企业微信/钉钉通道：Webhook URL，钉钉额外支持签名密钥
- 内置表单验证
- 编辑模式下可点击"测试发送"调用 `/api/notification-rules/{id}/test`
- 启用/停用开关

## 二、修改文件（后端）

### 2.1 `agentv2/models.py`

新增两个 ORM 模型：

```python
class NotificationRule(Base):
    __tablename__ = "notification_rules"
    id, name, event_type, channel, channel_config, is_enabled,
    created_by, created_at, updated_at

class Alert(Base):
    __tablename__ = "alerts"
    id, rule_id (FK), event_type, title, message, severity,
    source, is_sent, sent_at, created_at
```

### 2.2 `agentv2/crud.py`

通知规则 CRUD：
- `get_notification_rules(db)` — 按创建时间倒序
- `get_notification_rule(db, rule_id)` — 单条查询
- `create_notification_rule(db, rule_payload, created_by)` — 创建
- `update_notification_rule(db, rule_id, rule_payload)` — 更新
- `delete_notification_rule(db, rule_id)` — 删除

告警 CRUD：
- `get_alerts(db, limit, event_type)` — 列表查询，支持按事件类型筛选
- `create_alert(db, ...)` — 创建告警记录
- `get_alert_stats(db)` — 30 天统计（按事件类型 + 严重级别分组计数）
- `cleanup_old_alerts(db, retention_days=90)` — 90 天自动清理

数据集成：
- `_format_data_for_frontend()` 新增 `db_notification_rules` 参数，返回格式化的规则列表（含 JSON 解析后的 channel_config）
- `get_all_data()` 同步加载 `notification_rules`

### 2.3 `agentv2/services.py`

新增 `evaluate_and_alert(db, event_type, title, message, severity, source)` 函数：
- 查询所有匹配 event_type 且启用的通知规则
- 逐个调用 `deliver_notification()` 投递
- 返回触发规则数
- 当无匹配规则时，仍创建告警记录（静默记录，不投递），确保事件可追溯

### 2.4 `agentv2/core.py`

新增 Pydantic 模型 `NotificationRulePayload`：
```python
class NotificationRulePayload(BaseModel):
    id: str
    name: str
    event_type: str   # device_offline | brute_force | compliance_fail | system_error
    channel: str       # email | wechat_work | dingtalk
    channel_config: str  # JSON string
    is_enabled: bool = True
```

### 2.5 `agentv2/routers/admin.py`

新增 8 个端点，全部受 License 功能开关 `check_feature("alerting")` 保护：

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/api/notification-rules` | alert:manage | 获取所有通知规则 |
| POST | `/api/notification-rules` | alert:manage | 创建通知规则 |
| PUT | `/api/notification-rules/{id}` | alert:manage | 更新通知规则 |
| DELETE | `/api/notification-rules/{id}` | alert:manage | 删除通知规则 |
| POST | `/api/notification-rules/{id}/test` | alert:manage | 测试发送通知 |
| GET | `/api/alerts` | alert:manage | 获取告警历史（支持 event_type 筛选） |
| GET | `/api/alerts/stats` | alert:manage | 获取 30 天告警统计 |

未持有 `alerting` License 功能时，所有端点返回 403："告警通知功能需要企业版 License，请联系销售升级。"

## 四、架构关系

```
用户 → AdminPanel (告警通知标签页)
       ↓
     AlertRuleModal (创建/编辑规则)
       ↓
     alertSlice → apiFetch → /api/notification-rules (CRUD)
                            → /api/notification-rules/{id}/test (测试)

系统事件 → services.evaluate_and_alert()
              ↓
         notifications.deliver_notification()
              ↓
         send_email / send_wechat_work / send_dingtalk
              ↓
         crud.create_alert() → alerts 表记录
```

## 五、License 策略

- 功能开关：`check_feature("alerting")`
- 前端通过 `hasPermission(currentUser, ATOMIC_PERMISSIONS.ALERT_MANAGE)` 控制标签页可见性
- 后端所有 `/api/notification-rules` 和 `/api/alerts` 端点统一通过 `_require_alerting` 依赖检查
- 签发新 License 时需包含 `alerting` 功能项（`gen_license.py` 已默认包含）
