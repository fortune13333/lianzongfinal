### **(P2: 背景与痛点)**

首先是选题背景。在网络运维领域，我们一直面临着一个**“黑盒困境”**。
传统的网络配置管理非常依赖人工SSH登录设备操作。这种方式有三大痛点：
第一，**高风险**，一个微小的配置错误（比如配错了ACL）可能引发全网瘫痪。
第二，**不透明**，多人协作时，谁改了什么，很难实时知道。
第三，也是最关键的，**审计滞后**。传统的审计都是“事后诸葛亮”，出了事故才去查日志，无法在错误发生前进行拦截。

虽然市面上有Ansible这样的自动化工具，但它们侧重于“执行”，而在“治理”和“可信追溯”方面依然存在短板。

### **(P3: 目标与理念)**

针对这些问题，我设计了“链踪”系统。
我的核心设计理念是**“技术融合”**：
我利用**区块链**的不可篡改特性，解决“信任”问题；
利用**AI大模型**的推理能力，解决“治理”问题。
我的目标是构建一个从“事后追溯”进化到**“事前治理”**的智能平台。

---

### **(P4: 总体架构)**

这是系统的总体架构。
我采用了主流的**前后端分离**架构。前端使用 React 构建，后端使用 Python FastAPI。
为了保障企业数据安全，系统采用**本地代理（Local Agent）**模式部署，所有与网络设备的SSH连接都发生在内网，数据不出域。
这里特别提一下，为了适应不同的AI模型，我在后端设计了一个**“可插拔的AI驱动层”**，可以灵活切换 Gemini 或 讯飞星火等模型。

### **(P5: 功能模块)**
系统主要包含四大功能模块：**核心审计链**、**AI智能治理**、**实时交互终端**以及**自动化管理**。接下来我将重点介绍其中的核心创新点。

---

### **(P6: 创新点一 不可变审计链)**

第一个创新点是**基于哈希链的不可变记录**。
大家请看，我并没有使用笨重的公链，而是借鉴了区块链的核心数据结构。
每一次配置变更，系统都会生成一个“区块”。这个区块包含配置快照、操作员签名，最重要的是，它包含了**前一个区块的哈希值**。
这样就形成了一条环环相扣的链条。任何对历史数据库的篡改（比如删改日志），都会导致哈希校验失败，从而被系统识别。这为审计提供了**法证级**的可信度。

### **(P7: 创新点二 AI事前治理)**

第二个创新点，也是本系统的最大亮点，是**AI驱动的“事前治理”引擎**。
传统的审计是事后的，而“链踪”是事前的。
我设计了一个**“先审计，后执行”**的工作流。当用户提交配置时，请求会被挂起，系统会调用AI模型，结合预定义的“合规策略”（比如“禁止开启Telnet”），对配置进行审查。
**只有AI判定合规，变更才会被写入区块链**；如果违规，操作会被当场拦截。这真正实现了“防患于未然”。

### **(P8: 创新点三 安全交互终端)**

第三个创新点是**兼顾体验与安全的交互式终端**。
很多Web网管工具体验很差，但我利用WebSocket实现了原生CLI体验，支持Tab补全。
更重要的是，我在后端实现了一个**协议级的命令拦截器**。
当用户输入 `write memory` 这样涉及持久化的高危命令时，系统会在后端实时识别，并通过发送 `CTRL+U` 信号，物理清除设备端的输入行，强制用户必须通过系统的“保存并审计”流程来提交变更。
此外，我还设计了**“断连自动审计”**机制，即使用户强制关闭浏览器，系统也能自动捕获最后的配置状态，确保审计没有死角。

### **(P9: 创新点四 企业级特性)**

最后，为了提升实用性，我还实现了一系列企业级特性。
比如**原子化的权限管理**，我可以把“重启设备”这个单一权限下放给某个操作员，而不是给他管理员账号，符合最小权限原则。
还有**安全写入启动配置**流程，通过二次验证令牌，确保配置固化的安全性。
