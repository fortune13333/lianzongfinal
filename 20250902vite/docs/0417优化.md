本次对项目进行了全面代码审查，共发现并修复 **13 项**问题，按 P0/P1/P2 优先级分组。

## P0 安全修复（3 项）

### #1 JWT Token 改为 WebSocket 首帧传输

**问题**：WebSocket 建立连接时通过 URL 查询参数（`?token=...`）传递 JWT，Token 会被明文记录在服务端日志、Nginx 访问日志及浏览器历史记录中，存在信息泄露风险。

**修改文件**：
- `agentv2/websocket_handler.py`
- `src/components/InteractiveTerminal.tsx`

**后端变更**（`websocket_handler.py`）：
- 删除 `query_params.get("token")` 的取值逻辑
- 改为先 `await websocket.accept()` 建立连接
- 再通过 `asyncio.wait_for(websocket.receive_text(), timeout=5.0)` 接收第一条消息作为 Token
- 若 5 秒内未收到 Token，调用 `websocket.close(code=1008)` 关闭连接

**前端变更**（`InteractiveTerminal.tsx`）：
- 移除 WebSocket URL 中的 `?token=...` 参数
- 在 `ws.onopen` 回调中将 Token 作为第一条消息发送：
  ```ts
  ws.onopen = () => {
      const wsToken = sessionStorage.getItem('chaintrace_token') || '';
      ws.send(wsToken);
      ...
  };
  ```

**同步修复**：将 `ANSI_ESCAPE_RE` 正则常量从函数中间移动到文件顶部模块级别，消除"使用在前、定义在后"的代码顺序问题。

**修改文件**：`agentv2/routers/devices.py`

**变更**：在 `get_device_sessions` 函数签名中添加：
```python
actor: str = Depends(get_current_actor)
```

### #3 修复 Python 可变默认参数 Bug

**问题**：`agentv2/routers/topology.py` 中 `_build_response` 函数使用 `extra_node_ids: set = set()` 作为默认参数。Python 中可变对象默认参数在所有调用之间共享同一实例，会导致多次调用之间节点集合互相污染，产生难以复现的数据错误。

**修改文件**：`agentv2/routers/topology.py`

**变更**：
```python
# 修改前
def _build_response(links, all_managed_ids, extra_node_ids: set = set()):

# 修改后
def _build_response(links, all_managed_ids, extra_node_ids: Optional[set] = None):
    if extra_node_ids is None:
        extra_node_ids = set()
```

## P1 功能正确性修复（5 项）

### #4 401 响应改为事件驱动退出

**问题**：`apiFetch.ts` 遇到 401 时直接执行 `window.location.href = '/'` 硬跳转，会强制刷新页面，导致用户正在进行的 SSH 终端会话、配置编辑等操作全部丢失。

**修改文件**：
- `src/utils/apiFetch.ts`
- `src/App.tsx`

**`apiFetch.ts` 变更**：将硬跳转替换为派发自定义事件：
```ts
window.dispatchEvent(new CustomEvent('chaintrace:unauthorized'));
```

**`App.tsx` 变更**：添加事件监听器，事件触发时调用 Zustand 的 `logout()` 进行软退出：
```ts
React.useEffect(() => {
  const handler = () => logout();
  window.addEventListener('chaintrace:unauthorized', handler);
  return () => window.removeEventListener('chaintrace:unauthorized', handler);
}, [logout]);
```

### #5 `saveSessionAndAudit` 错误步骤状态修正

**问题**：`sessionSlice.ts` 的 `catch` 块中，无论是用户主动取消（`AbortError`）还是网络错误，都将第一步标记为 `error` 状态，导致取消操作时也显示红色错误提示，误导用户。

**修改文件**：`src/store/slices/sessionSlice.ts`

**变更**：区分 `AbortError` 与其他错误：
- `AbortError`（用户取消 / 超时）→ 三步全部保持 `pending`（灰色）
- 其他网络/服务器错误 → 第 1 步 `error`，第 2、3 步 `pending`

---

### #6 `get_all_data` 限制区块加载量

**问题**：`crud.py` 的 `get_all_data` 通过 `joinedload(Device.blocks)` 在每次轮询时加载所有设备的全部区块历史，设备数据量大时会导致严重的内存占用和响应延迟（每 5 秒一次的轮询尤为突出）。

**修改文件**：`agentv2/crud.py`

**变更**：
- 拆分为两个独立查询：先查所有设备（仅 joinedload policies），再单独按 `(device_id, index DESC)` 排序查询 Block
- Python 层对每个设备最多保留最新 **50** 条区块
- `_format_data_for_frontend` 接受新参数 `blocks_by_device: Optional[Dict[str, List]]`，不再依赖 ORM 关系属性 `device.blocks`

### #7 Device.type 后端枚举验证

**问题**：`DevicePayload` 中 `type` 字段为 `str` 类型，客户端可传入任意字符串（如 `"hacker"`），缺乏服务端校验，可能导致前端渲染异常或数据不一致。

**修改文件**：`agentv2/core.py`

**变更**：
```python
from typing import Literal

class DevicePayload(BaseModel):
    type: Literal['Router', 'Switch', 'Firewall']
    # 其余字段不变
```
非法值将由 Pydantic 自动返回 422 Unprocessable Entity。

### #8 定时任务模拟模式状态修正

**问题**：`scheduler.py` 中，模拟模式下任务执行被错误计入 `success` 计数，导致任务状态始终显示 `success`，无法区分"真正成功"和"模拟执行"，掩盖了系统实际工作状态。

**修改文件**：`agentv2/scheduler.py`

**变更**：引入独立的 `simulated` 计数器：
```python
simulated = 0
# 模拟模式时：
simulated += 1
continue  # 不计入 success

# 最终状态判断：
if errors > 0:
    status = "error"
elif simulated > 0 and success == 0:
    status = "simulated"
else:
    status = "success"
```

## P2 代码质量改进（5 项）

### #9 代码分割（Code Splitting）

**问题**：`TopologyView` 和 `TerminalContainer` 是体积较大的组件（包含 `@xyflow/react`、`xterm.js` 等重型依赖），在应用初始加载时同步打包，导致首屏 JS bundle 体积过大，影响加载性能。

**修改文件**：`src/App.tsx`

**变更**：改用 `React.lazy` 懒加载，配合 `<Suspense>` 提供加载状态：
```tsx
const TopologyView = React.lazy(() => import('./components/TopologyView'));
const TerminalContainer = React.lazy(() => import('./components/TerminalContainer'));

// 使用处：
<Suspense fallback={<div className="flex-grow flex items-center justify-center"><Loader /></div>}>
  <TerminalContainer />
</Suspense>
```

### #10 `window.confirm` 替换为 ConfirmationModal

**问题**：删除设备、删除脚本、删除定时任务、重置数据等危险操作使用原生 `window.confirm` 弹窗，样式与应用整体风格不统一，且阻塞主线程。

**修改文件**：
- `src/store/storeTypes.ts`
- `src/store/slices/deviceSlice.ts`
- `src/store/slices/dataSlice.ts`
- `src/App.tsx`
- `src/components/admin/AdminPanel.tsx`

**方案**：Zustand slice 中添加"等待确认"状态，由 UI 层渲染 `ConfirmationModal`。

**`storeTypes.ts`** 新增类型：
- `DeviceSlice`：`deleteConfirmDeviceId: string | null`、`promptDeleteDevice`、`cancelDeleteDevice`
- `DataSlice`：`isResetConfirmOpen: boolean`、`promptResetData`、`cancelResetData`

**`App.tsx`**：订阅上述状态，渲染对应的 `<ConfirmationModal>`（删除设备 / 重置数据各一个）

**`AdminPanel.tsx`**：
- 将 `DeletionTarget` 联合类型扩展，新增 `script` 和 `scheduledTask` 两种类型
- 将两处 `window.confirm(...)` 替换为 `setDeletionTarget({ type: 'script'/'scheduledTask', item })`
- `executeDelete` 和 `getDeletionModalContent` 对应处理新类型

### #12 修复任务序列化字段不一致

**问题**：`routers/tasks.py` 中的 `_task_to_dict` 函数缺少 `created_at` 字段，而 `crud._format_data_for_frontend` 通过 `/api/data` 返回数据时包含该字段，导致通过 `/api/scheduled-tasks` 列表接口获取的数据与全量数据接口不一致，前端可能因字段缺失出现显示异常。

**修改文件**：`agentv2/routers/tasks.py`

**变更**：在 `_task_to_dict` 中补充字段：
```python
"created_at": task.created_at.isoformat().replace('+00:00', 'Z') if task.created_at else None,
```

> `routers/scripts.py` 的脚本序列化已包含 `created_at`，无需修改。

### #13 配置搜索改用 SQL LIKE 预过滤

**问题**：`routers/admin.py` 的配置全文搜索端点 `/api/search` 先从数据库加载**全部** Block 记录，再在 Python 层逐条过滤匹配关键词。当设备历史配置数量庞大时，会产生大量无意义的 DB → Python 数据传输，响应时间随数据量线性增长。

**修改文件**：`agentv2/routers/admin.py`

**变更**：在 Python 过滤前，先在 SQL 层用 `ILIKE` 预筛选，并加 `limit` 限制传输量：
```python
query = db.query(models.Block).filter(models.Block.data.ilike(f'%{q}%'))
if device_id:
    query = query.filter(models.Block.device_id == device_id)
blocks = query.limit(limit * 10).all()  # 取 10 倍 limit，Python 层精确过滤后裁到 limit
```

**效果**：只有包含关键词的区块才会被 Python 处理，大数据量下查询性能显著提升。
