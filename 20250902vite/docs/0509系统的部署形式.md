链踪是纯私有化、单服务器部署，客户端只需要一个浏览器。

物理部署架构

【客户内网】

  工程师的电脑          运维服务器（部署链踪）          网络设备
  ┌──────────┐          ┌─────────────────────┐          ┌──────────┐
  │  浏览器  │ ←HTTPS→ │   链踪（Docker）     │ ←SSH→   │ 路由器   │
  └──────────┘          │                     │          └──────────┘
                        │  ┌───────────────┐  │          ┌──────────┐
  工程师的电脑          │  │  前端（静态）  │  │ ←SSH→   │ 交换机   │
  ┌──────────┐          │  │  FastAPI 后端  │  │          └──────────┘
  │  浏览器  │ ←HTTPS→ │  │  PostgreSQL    │  │          ┌──────────┐
  └──────────┘          │  │  nginx（SSL）  │  │ ←SSH→   │ 防火墙   │
                        │  └───────────────┘  │          └──────────┘
  管理员的电脑          └─────────────────────┘
  ┌──────────┐
  │  浏览器  │
  └──────────┘
核心要点：

所有链踪组件运行在客户自己的一台 Linux 服务器上，数据不出内网
工程师不需要安装任何客户端软件，打开浏览器即可使用
链踪服务器通过 SSH 访问网络设备，工程师从不直接 SSH 登录设备
三种部署模式
模式	适用场景	启动方式
开发模式	本地调试	start.bat（前后端分开跑）
Docker 单机（HTTP）	内网试用、小规模生产	docker compose up
Docker 单机（HTTPS）	正式生产、等保要求	docker compose --profile https up
正式交付给客户的都是 Docker 单机 HTTPS 模式，整个系统只需要一台服务器、一条命令启动。

Docker 内部组件关系

docker compose up 启动后，内部有 3–4 个容器：

┌─────────────────────────────────────────┐
│              docker compose             │
│                                         │
│  ┌─────────┐    ┌──────────────────┐   │
│  │  nginx  │───▶│  chaintrace      │   │
│  │ :443    │    │  (FastAPI+前端静态│   │
│  └─────────┘    │   文件)  :8000   │   │
│  （仅 https     └────────┬─────────┘   │
│    profile）             │              │
│                  ┌───────▼──────────┐  │
│                  │   PostgreSQL     │  │
│                  │   :5432 (内部)   │  │
│                  └──────────────────┘  │
└─────────────────────────────────────────┘
前端静态文件（npm run build 产物）在构建阶段就打包进了后端镜像，所以对外只有一个服务端口。

对客户的部署要求
项目	最低要求
服务器 OS	Ubuntu 22.04 / Debian 12 / CentOS 8+
硬件	2 核 4G 内存，50G 硬盘（小规模）
网络	服务器能 SSH（22端口）到达所有被管设备
软件	仅需安装 Docker + Docker Compose
客户端	无需安装，Chrome/Edge/Firefox 即可
这也是当前商业化的一个加分点——交付物就是一个 docker-compose.yml 文件，IT 运维人员按文档操作 30 分钟之内可以跑起来。
