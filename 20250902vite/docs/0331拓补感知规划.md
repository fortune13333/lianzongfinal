# 伪代码示意
def perform_discover_topology(device_id: str) -> list[dict]:
    device_info = get_device_info(device_id)
    with ConnectHandler(**device_info) as conn:
        cdp_output = conn.send_command("show cdp neighbors detail")
        lldp_output = conn.send_command("show lldp neighbors detail")
    # 交给 AI 解析为结构化数据
    return parse_neighbors_with_ai(cdp_output or lldp_output)

Device ID: SW01-SFO
  IP address: 10.0.0.2
  Platform: Cisco WS-C2960
  Interface: GigabitEthernet0/0,  Port ID (outgoing port): GigabitEthernet0/1
输出（JSON）：


{

  "local_port": "GigabitEthernet0/0",

注意事项：

模拟模式：用种子数据中的 3 台设备生成 mock 拓扑，保证演示效果

四、变更影响分析
这是最有价值的部分，也是难度最高的部分。

流程设计：


用户准备推送配置
        ↓
系统自动触发影响分析
        ↓
后端：
  1. 从拓扑 DB 查找目标设备的 1~2 跳邻居
  2. 从区块链获取邻居设备的最新配置
  3. 将【目标设备当前配置 + 邻居配置 + 拟变更内容 + 拓扑关系】发给 AI
        ↓
AI 返回影响报告（示例）：
  ⚠️  高风险：修改 RTR01-NYC 的 OSPF area 0 配置
      - SW01-SFO 的 Fa0/1 接口参与同一 OSPF 区域，可能导致邻居关系重置
      - 预计影响时间窗口：30-90 秒路由收敛
  ℹ️  低风险：修改 hostname 命令
      - 无联动风险
        ↓
用户查看报告，确认或取消变更
与现有代码的集成点：

perform_pre_deployment_check()：在此基础上增加拓扑感知上下文
/api/ai/check-config：扩展 prompt，加入邻居信息
push_config_to_device 和 bulk_deploy_template：推送前调用影响分析接口
AI 能力边界（需如实告知）：

✅ 能识别：路由协议邻居关系、ACL/防火墙策略冲突、接口配置不一致
⚠️ 需谨慎：复杂多跳传播分析，准确性依赖 AI 模型质量
❌ 不可靠：实时流量影响预测（需要 SNMP/NetFlow 数据支持）

七、建议开发顺序
先做拓扑发现（不依赖 AI），用结构化 prompt 让 AI 解析 CDP/LLDP 输出
再做前端可视化，此阶段可以用硬编码 mock 数据快速验证 UI 效果
最后做影响分析，在前两个基础上叠加 AI 推理层
总结： 这个功能与项目现有架构（Netmiko SSH、AI 代理、区块链配置历史）高度契合，不需要引入颠覆性的新技术。最大的变量是 AI 影响分析的质量——这取决于 AI 模型本身，但即使分析不完美，"提示用户注意邻居设备"本身就已经有很高的价值。

如果你想开始实现，我建议从**阶段一（后端发现 API）**开始，确认数据结构后再推进前端。
