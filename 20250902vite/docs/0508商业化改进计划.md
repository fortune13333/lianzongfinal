# 商业化改进计划书

**版本**：v1.0  
**日期**：2026-05-08  
**背景**：项目已完成毕业设计阶段，当前目标为商业化落地，主要切入场景为**中小企业 / 政企单位等保合规需求**。

---

## 一、核心问题与优先级判断

经代码审查，商业化路上存在三个**硬缺口**，不解决则无法进入企业采购流程：

| 优先级 | 问题 | 当前状态 | 商业影响 |
|---|---|---|---|
| P0 | 数据库无法支撑生产规模 | SQLite 单文件，无 HA | 无法回答"能撑多少并发"的问题 |
| P0 | 没有企业认证集成 | 仅用户名/密码，无 LDAP/AD/SSO | 进不了企业 IT 采购流程 |
| P1 | 缺少商业产品的交付闭环 | 无 PDF 报告、无告警、无 License 机制 | 客户无法用产品完成等保验收 |

### 2.1 数据库迁移：SQLite → PostgreSQL

**为什么是第一优先**：后续所有功能（多租户、并发、HA）都建立在这个基础上。先迁移，后续才能推进。

#### 改动范围

**`agentv2/database.py`**（核心改动）：
```python
# 现状：
DATABASE_URL = "sqlite:///./chaintrace.db"

# 目标：
import os
DATABASE_URL = os.environ.get(
    "DATABASE_URL",
    "postgresql+psycopg2://chaintrace:password@localhost:5432/chaintrace"
)
```
同步移除 `connect_args={"check_same_thread": False}`（SQLite 专属）。

**`requirements.txt`** 新增：
```
psycopg2-binary>=2.9
```

**`docker-compose.yml`** 新增 PostgreSQL 服务：
```yaml
postgres:
  image: postgres:16-alpine
  environment:
    POSTGRES_DB: chaintrace
    POSTGRES_USER: chaintrace
    POSTGRES_PASSWORD: ${DB_PASSWORD}
  volumes:
    - pgdata:/var/lib/postgresql/data
  ports:
    - "5432:5432"
```

**Alembic 迁移注意事项**：
- `004_add_block_unique_constraint.py` 中的 `CREATE UNIQUE INDEX` 语法在 PostgreSQL 上兼容，无需改动
- 但 `topology.py` 中用 `op.execute()` 写的原始 SQL 需要验证 PostgreSQL 兼容性
- `blocks.data` 列目前是 `TEXT` 存 JSON 字符串；迁到 PostgreSQL 后可改为原生 `JSONB` 类型，搜索性能大幅提升（同步影响 `#13` 的 `ilike` 搜索）

**`agentv2/models.py`** 可选优化（迁 PostgreSQL 后）：
```python
from sqlalchemy import JSON
# blocks.data: TEXT → JSON/JSONB（需要新 alembic 迁移）
```

**验证方式**：`cd agentv2 && alembic upgrade head`，再跑全部 76 个后端测试。

### 2.2 LDAP / Active Directory 认证接入

**为什么重要**：大多数企业 IT 环境强制要求所有系统接入 AD/LDAP 统一身份认证，这是采购合同的前置条件。

#### 实现方案

采用"LDAP 验证 + 本地用户补全"混合模式：
- 用户在 LDAP 中认证身份（密码验证交给 AD）
- 首次成功登录后，系统自动在本地 `users` 表创建账户记录（用于权限分配）
- 本地账户只存权限配置，不存密码

**`requirements.txt`** 新增：
```
python-ldap3>=2.9
```

**`agentv2/core.py`** 新增配置模型：
```python
class LDAPConfig(BaseModel):
    enabled: bool = False
    server: str = ""           # ldap://192.168.1.10
    port: int = 389
    base_dn: str = ""          # DC=company,DC=com
    bind_dn: str = ""          # CN=svc_chaintrace,OU=ServiceAccounts,...
    bind_password: str = ""
    user_search_filter: str = "(sAMAccountName={username})"
    use_ssl: bool = False
```

**`agentv2/auth_deps.py`** 新增 LDAP 验证函数：
```python
def verify_ldap(username: str, password: str, config: LDAPConfig) -> bool:
    from ldap3 import Server, Connection, ALL, NTLM
    server = Server(config.server, port=config.port, get_info=ALL)
    search_filter = config.user_search_filter.format(username=username)
    # 先用 bind_dn 搜索用户 DN，再用用户凭证二次绑定验证
    ...
```

**`agentv2/routers/auth.py`** 登录逻辑改造：
```python
@router.post("/api/login")
def login(payload: LoginPayload, db: Session = Depends(get_db)):
    ldap_cfg = get_ldap_config(db)  # 从 settings 表读
    if ldap_cfg.enabled:
        if not verify_ldap(payload.username, payload.password, ldap_cfg):
            raise HTTPException(401, "LDAP 认证失败")
        user = get_or_create_ldap_user(db, payload.username)  # 自动建本地账户
    else:
        user = verify_local_password(db, payload.username, payload.password)
    # 后续 JWT 签发流程不变
```

**前端** `SettingsModal.tsx` 新增 LDAP 配置面板（管理员可见），包含服务器地址、Base DN、绑定账号等字段，调用新增的 `PUT /api/settings/ldap` 端点。

### 2.3 合规审计报告 PDF 导出

**为什么重要**：等保验收的直接交付物就是审计报告。客户用你的产品，最终要打印一份报告给审查机构看。

#### 实现方案

**`requirements.txt`** 新增：
```
weasyprint>=60.0   # HTML → PDF，支持中文，无需 wkhtmltopdf
```

**`agentv2/routers/admin.py`** 新增端点：
```python
GET /api/report/audit?start=2026-01-01&end=2026-05-01&device_id=OPTIONAL
GET /api/report/compliance?device_id=REQUIRED&version=OPTIONAL
```

**报告内容**（使用 Jinja2 渲染 HTML 后转 PDF）：
- 封面：系统名称、报告生成时间、生成人、时间范围
- 设备配置变更历史表：时间、操作人、变更类型、哈希值
- 合规检查结果汇总：策略命中情况、违规记录
- 区块链完整性验证结果（哈希链校验通过/失败）
- 附录：关键配置快照

**前端**：Dashboard 工具栏新增"导出报告"按钮，弹出日期范围选择器，调用接口后触发文件下载。

### 3.1 告警与通知系统

**当前状态**：系统是完全被动的，只能人工查询。企业运维的核心诉求是"出问题了自动通知我"。

#### 触发条件（按优先级）

| 触发事件 | 告警级别 |
|---|---|
| 合规检查不通过（severity: critical） | 紧急 |
| 设备配置被非计划变更（非定时任务触发） | 高 |
| 设备 SSH 连接状态从 online → offline | 中 |
| 定时任务执行失败 | 中 |
| 用户连续登录失败（可能被暴力破解） | 高 |

#### 通知渠道实现

**`agentv2/services.py`** 新增通知模块：
```python

        # 企业微信 Webhook（POST JSON）

```

**`agentv2/models.py`** 新增 `notification_rules` 表：
```python
class NotificationRule(Base):
```

### 3.3 产品授权（License）机制

**为什么需要**：没有 License 机制，产品无法定价，无法按特性分层收费，无法防止无限制扩散。

#### 设计方案（离线 License，适合私有化部署）

**License 内容**（JSON，RSA 私钥签名）：
```json
{
  "customer": "XX科技有限公司",
  "max_devices": 50,
  "features": ["ldap", "pdf_report", "notification"],
  "expires_at": "2027-05-08",
  "issued_at": "2026-05-08"
}
```

**`agentv2/core.py`** 新增 License 验证：
```python
def verify_license(license_file: str, public_key: str) -> LicenseInfo:
    # RSA 公钥验签 → 解析 JSON → 检查 expires_at、max_devices
```

**`agentv2/agent.py`** 启动时验证：
- License 无效 → 降级为"社区版"（限制设备数量 ≤ 5，无 LDAP/PDF/告警功能）
- License 有效 → 正常启动，功能按 `features` 字段解锁

**分层定价建议**：
| 版本 | 设备上限 | 功能 |
|---|---|---|
| 社区版（免费） | 5 台 | 核心审计、终端、AI 检查 |
| 专业版 | 50 台 | + LDAP、PDF 报告 |
| 企业版 | 无限制 | + 告警通知、多租户、优先支持 |

## 四、第三阶段：扩大覆盖面（6–12 个月后）

**目标**：从单客户工具升级为平台，支撑 MSSP 场景和更大规模部署。

---

### 4.1 多租户架构

**当前问题**：所有数据（设备、区块、审计日志）在同一个命名空间，无法服务多个相互隔离的客户组织。

#### 改动范围（工程量大，需专项规划）

所有主要 ORM 模型添加 `org_id` 外键：
```python
class Organization(Base):
    id: str (PK)
    name: str
    license_key: str
    
# Device, Block, User, AuditLog, Policy... 全部加 org_id FK
```

所有查询在 SQL 层加 `filter(Model.org_id == current_org_id)`。

JWT Payload 中嵌入 `org_id`，`get_current_actor` 同时返回 actor 和 org_id。

**这是工程量最大的改动，建议在有第一批稳定客户后再推进，不要过早投入。**

---

### 4.2 华为 / 华三设备可靠适配

**当前问题**：Netmiko 的 `send_command` 在华为 VRP 和华三 Comware 上输出格式差异大，`get_running_config` 等命令名称不同，CDP/LLDP 解析正则在国产设备上失效。

#### 改动范围

**`agentv2/services.py`** 按 `device_type` 分派命令：
```python
SHOW_CONFIG_COMMANDS = {
    "cisco_ios":   "show running-config",
    "cisco_nxos":  "show running-config",
    "huawei":      "display current-configuration",
    "hp_comware":  "display current-configuration",
}
```

LLDP/CDP 解析器 `agentv2/routers/topology.py` 补充华为 `display lldp neighbor` 和华三 `display lldp neighbor-information` 的输出格式解析。

---

### 4.3 SNMP Trap 接收 → 自动触发审计

**`requirements.txt`** 新增：`pysnmp`

新增后台线程监听 UDP 162 端口，接收设备发来的 `configChange` Trap（OID: `1.3.6.1.4.1.9.9.43.2.0.1`），自动触发对应设备的配置拉取和区块链写入。

这样可以实现**零延迟的配置变更感知**，不依赖 5 秒轮询。

## 五、非功能性改进（贯穿全程）

| 项目 | 具体内容 | 阶段 |
|---|---|---|
| **速率限制** | 当前 rate limiter 是进程内 dict，多 Worker 下失效。改用 Redis 或 PostgreSQL 表存储计数 | 第一阶段 |
| **JWT 吊销** | 当前 JWT 无法强制失效（如管理员修改用户权限后旧 Token 仍有效）。引入 `token_version` 字段，每次权限变更自增，JWT 验证时比对 | 第一阶段 |
| **WebSocket 鉴权** | 已改为首帧传 Token，但 Token 过期后 WS 连接不会自动断开。需在心跳逻辑中定期验证 Token 有效性 | 第二阶段 |
| **API 文档** | FastAPI 自带 OpenAPI，在 `agent.py` 中将 `docs_url` 设为仅管理员可访问，对外提供 Swagger UI 方便客户集成 | 第一阶段 |
| **部署文档** | 面向 IT 管理员的一键部署指南（Docker Compose），包含 PostgreSQL 初始化、SSL 证书配置、License 激活 | 第一阶段 |

## 七、最重要的一件事

**在第一阶段完成后，立刻去找 1–2 个真实客户做免费试点**，哪怕是朋友公司的内网环境。

真实客户会告诉你：
- 他们实际卡在哪个功能（大概率不是你认为重要的）
- 他们的 AD 域结构是否与你的 LDAP 实现兼容
- PDF 报告里他们真正需要哪些字段

没有真实反馈，后续的所有开发方向都是猜测。
