# 答辩常见问题与回答策略库 (Q&A)

在答辩中，老师通常会关注**必要性、技术实现深度、安全性和实用性**。以下是针对“链踪”项目的高频预测问题及回答策略。

---

### **一、 关于选题与价值**

**Q1: 既然已经有 Git 或者 Ansible 这样的工具了，为什么还要做一个“区块链”审计系统？是不是为了蹭区块链的热度？**

*   **回答策略**: 承认现有工具的价值，但强调“链踪”独特的**“不可篡改性”**和**“治理属性”**。
*   **参考话术**:
    > “这是一个很好的问题。Ansible 确实擅长配置的分发和部署，Git 擅长版本控制。但它们在**‘审计可信度’**上存在短板。
    > 传统的数据库或 Git 记录，拥有管理员权限的人是可以悄无声息地修改历史记录的（比如 `git push --force`）。而我引入哈希链结构，目的是为了解决**‘谁来审计审计员’**的问题。一旦区块生成，其哈希值环环相扣，任何篡改都会导致链断裂。
    > 此外，‘链踪’侧重于**AI驱动的事前治理**（拦截违规），这是 Ansible 原生不具备的。”

**Q2: 你这个区块链是去中心化的吗？如果不是，那还有什么意义？**

*   **回答策略**: 明确指出是**“中心化应用，借鉴区块链思想”**，不要硬说是公链。
*   **参考话术**:
    > “我的系统**不是**去中心化的公链（如比特币），不需要挖矿。
    > 我借鉴的是区块链核心的**‘哈希链（Hash Chain）’数据结构**。在企业内部管理场景中，去中心化会带来不必要的性能开销和管理复杂度。
    > 我采用中心化部署 + 哈希链结构，既保证了数据的**不可篡改性**（密码学保证），又保证了系统的高性能和易维护性。这是一种务实的技术选型。”

### **二、 关于 AI 技术实现**

**Q3: AI 可能会产生幻觉（胡说八道），你怎么保证它生成的配置或审计结果是准确的？**

*   **回答策略**: 强调**“指令工程”**、**“人机协同”**和**“多重防线”**。
*   **参考话术**:
    > “AI 幻觉确实是 LLM 的通病。我在设计中采取了三层措施来规避风险：
    > 1.  **精细的 Prompt Engineering（指令工程）**：我不是简单地问 AI，而是提供了非常详尽的上下文（设备类型、当前配置、策略规则），并强制要求它返回结构化的 JSON 数据，限制了它的发挥空间。
    > 2.  **事后校验**：AI 生成的命令（如回滚方案）并不是直接执行，而是呈现给管理员，由人工进行最终确认（Human-in-the-loop）。
    > 3.  **兜底机制**：对于高风险操作（如写入启动配置），系统强制要求二次验证和权限检查，不完全依赖 AI 的判断。”

**Q4: 你的 AI 是怎么对接的？如果网络不通或者 Key 泄露了怎么办？**

*   **回答策略**: 展示**“后端代理架构”**的安全性。
*   **参考话术**:
    > “在安全性方面，我采用了**前后端分离的代理模式**。
    > 所有的 AI API Key 都存储在后端服务器的 `config.ini` 配置文件中，前端浏览器是接触不到 Key 的，这避免了泄露风险。
    > 另外，我的后端架构是**模块化**的（AI Drivers），支持热切换。如果 Google Gemini 连不上，我可以瞬间切换到国内的讯飞星火模型，代码已经做好了适配。”

### **三、 关于核心功能实现**

**Q5: 你的“实时命令拦截”具体是怎么实现的？如果我手速很快，能不能绕过？**

*   **回答策略**: 解释**“协议级拦截”**，而非前端拦截。
*   **参考话术**:
    > “我的拦截机制不是在前端做的（前端拦截很容易被绕过），而是在**后端 WebSocket 流**中处理的。
    > 后端维护了一个行缓冲区。当用户按下回车时，后端会先在内存中检查命令是否违规。
    > 如果违规，后端**根本不会**把换行符发给设备，而是发送一个 `CTRL+U` 信号。这是一个标准的终端控制符，作用是**清空当前行**。
    > 所以，无论用户手速多快，违规命令在物理上根本没有机会被发送给设备的解释器执行。”

**Q6: 如果用户直接强制关闭浏览器，是不是就能逃避审计了？**

*   **回答策略**: 引出**“断连自动审计”**这个亮点功能。
*   **参考话术**:
    > “这正是我在 V8.2 版本重点解决的问题。
    > 我设计了一个**‘断连自动审计’**机制。后端会跟踪会话状态，如果发现会话是‘脏’的（即用户修改过配置）且连接异常断开，后端会**自动触发**一次审计流程。
    > 它会重新连接设备，抓取当前的运行配置，并强制生成一个‘自动快照’区块。这样就形成了一个**审计闭环**，没有任何操作能逃逸记录。”

**Q7: 你的回滚功能是全量覆盖还是增量修改？**

*   **回答策略**: 解释**“全量覆盖”**的确定性优势。
*   **参考话术**:
    > “我采用的是**全量覆盖**策略。虽然增量修改（只下发 No 命令）看起来更优雅，但在复杂的网络环境中，计算精准的差异命令极易出错，可能导致设备处于中间状态。
    > 全量推送目标版本的配置，虽然‘笨’一点，但它具有**确定性**（Idempotency），能确保设备状态 100% 回到我们要的样子，这在应急恢复场景下是最安全的。”

### **四、 综合类**

**Q8: 项目最大的难点是什么？你是怎么解决的？**

*   **回答策略**: 挑一个技术含量高的点，比如**“交互式终端的实时性与安全性的平衡”**。
*   **参考话术**:
    > “最大的难点在于如何在提供流畅的 SSH 体验（比如 Tab 补全）的同时，还能拦截高危命令。
    > Tab 补全需要实时转发按键给设备，而拦截需要阻止命令执行。这是一个矛盾。
    > 我通过深入研究 SSH 和终端协议，最终采用了**‘实时转发按键 + 回车时缓冲检查 + 协议级清除’**的方案，既保留了原生体验，又守住了安全底线。”

**Q9: 你的系统能支持多少台设备？性能瓶颈在哪里？**

*   **回答策略**: 实事求是，指出 SQLite 的局限和未来的优化方向。
*   **参考话术**:
    > “目前的版本使用的是 SQLite 数据库，对于中小规模（几十台到上百台设备）的局域网管理是完全够用的。
    > 如果要扩展到大规模企业应用，瓶颈可能在于**数据库的并发写入**和**AI API 的调用延迟**。
    > 在未来的规划中，我已经预留了接口，可以将数据库无缝迁移到 PostgreSQL，并引入消息队列（如 Celery）来异步处理耗时的 AI 任务。”
