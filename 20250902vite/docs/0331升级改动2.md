## 一、后端路由拆分（P1）

### 背景

原 `agentv2/api_routes.py` 包含 877 行代码，将认证、设备、区块链、模板、脚本、定时任务、用户管理等全部混合在一个文件中，可读性和可维护性差。

### 改动方案

将所有路由按业务域拆分到 `agentv2/routers/` 目录，`api_routes.py` 仅保留 `include_router` 聚合调用。

### 新增 / 修改文件

#### `agentv2/api_routes.py`（重写，由 877 行缩减为 12 行）

```python
# 仅保留路由聚合
from routers import auth, devices, blockchain, templates, scripts, tasks, admin

router = APIRouter()
router.include_router(auth.router)
router.include_router(devices.router)
# ...（共 7 个 include_router）
```

#### `agentv2/routers/__init__.py`（已存在，保持不变）

包初始化文件，仅含注释。

#### `agentv2/routers/auth.py`（新建，~35 行）

- `POST /api/login` — 登录接口
- 从 `auth_deps` 导入 `check_login_rate_limit`

#### `agentv2/routers/devices.py`（已在上次改动中新建，本次确认）

- `GET /api/data` — 获取所有数据（含 netmiko_device_type 富化）
- `POST /api/reset` — 重置数据
- `POST/PUT/DELETE /api/devices` / `/api/devices/{id}` — 设备 CRUD
- `GET /api/device/{id}/running-config` — 获取设备运行配置
- `POST /api/device/{id}/push_config` — 非交互推送配置（调用 `services.perform_push_config`，**不直接导入 netmiko**）
- `GET /api/devices/poll-status` — 并发 TCP 22 端口检测
- `GET /api/devices/{id}/export` — 导出设备配置历史
- `GET/POST/DELETE /api/sessions/{id}` — 协作会话管理
- `POST /api/devices/{id}/write-startup` — 写入启动配置
- `POST /api/write-tokens` — 生成一次性写入令牌

#### `agentv2/routers/blockchain.py`（新建，~80 行）

- `POST /api/sessions/{device_id}/save` — 保存会话并审计（含 Race Condition 修复：先标记 session 为 clean，再拉取配置）
- `POST /api/blockchains/{device_id}` — 手动提交新区块
- `POST /api/blockchains/{device_id}/rollback` — 回滚到指定版本
- 不直接导入 netmiko，通过 `services.py` 的已有函数完成操作

#### `agentv2/routers/templates.py`（新建，~130 行）

- `POST/PUT/DELETE /api/templates` / `/api/templates/{id}` — 配置模板 CRUD
- `POST /api/bulk-deploy` — 批量部署（Jinja2 渲染 + 合规检查 + **调用 `services.perform_execute_ssh_config`**，不直接使用 `ConnectHandler`）

#### `agentv2/routers/scripts.py`（新建，~140 行）

- `GET/POST/PUT/DELETE /api/scripts` / `/api/scripts/{id}` — 脚本 CRUD
- `POST /api/scripts/{id}/execute` — 批量执行脚本（Jinja2 渲染 + 命令拦截检查 + AI 合规检查 + **调用 `services.perform_execute_ssh_config`**）

#### `agentv2/routers/tasks.py`（新建，~75 行）

- `GET/POST/PUT/DELETE /api/scheduled-tasks` / `/api/scheduled-tasks/{id}` — 定时任务 CRUD
- 提取了 `_task_to_dict()` 辅助函数，避免重复序列化逻辑

#### `agentv2/routers/admin.py`（新建，~200 行）

包含以下路由：
- `GET /api/health` — 健康检查
- `GET/POST/PUT/DELETE /api/users` / `/api/users/{id}` — 用户管理
- `POST/PUT/DELETE /api/policies` / `/api/policies/{id}` — 策略管理
- `PUT /api/settings/ai` — AI 设置
- `GET /api/backup` — 全量系统备份
- `GET /api/search` — 配置全文搜索
- `POST /api/ai/generate-command` — AI 命令生成代理
- `POST /api/ai/check-config` — AI 配置体检代理

### 关键设计原则

1. **路由文件不导入 netmiko**：所有 SSH 操作通过 `services.py` 的 `perform_push_config` / `perform_execute_ssh_config` 完成。
2. **权限统一**：所有路由通过 `auth_deps.get_current_actor` 或 `auth_deps.require_permission(...)` 鉴权。
3. **无业务逻辑泄漏**：路由文件只做参数校验、调用 crud/services、返回响应，不包含复杂业务逻辑。

## 二、前端 Store 拆分（P1）

### 背景

原 `src/store/useStore.ts` 包含 627 行代码，将认证、数据获取、设备操作、UI 状态、会话管理、脚本、定时任务全部混合在一个大文件中。

### 改动方案

使用 Zustand 的 `StateCreator` 模式将 store 拆分为 7 个 slice，并通过 `storeTypes.ts` 统一管理类型定义，避免循环导入。

### 新增文件

#### `src/store/storeTypes.ts`（新建）

定义所有 slice 的接口类型及 `FullStore` 联合类型：

```typescript
export interface AuthSlice { ... }
export interface DataSlice { ... }
export interface DeviceSlice { ... }
export interface UiSlice { ... }
export interface SessionSlice { ... }
export interface ScriptSlice { ... }
export interface TaskSlice { ... }

// 供所有 slice 文件作为 StateCreator 的泛型参数
export type FullStore = AuthSlice & DataSlice & DeviceSlice & UiSlice & SessionSlice & ScriptSlice & TaskSlice;
```

#### `src/store/slices/authSlice.ts`（新建）

状态：`currentUser`, `agentMode`, `aiStatus`
操作：`init`, `login`, `logout`

- `login` 会调用 `get().updateSettings`、`get().addAgentUrlToHistory`、`get().startStatusPolling`（跨 slice 调用，通过 `FullStore` 类型安全实现）
- `logout` 会调用 `get().stopStatusPolling`

#### `src/store/slices/dataSlice.ts`（新建）

状态：`devices`, `allUsers`, `auditLog`, `templates`, `policies`, `deploymentHistory`, `writeTokens`, `scripts`, `scheduledTasks`, `blockchains`, `isLoading`, `backendSettings`
操作：`fetchData`, `resetData`

- `fetchData` 包含用户数据刷新逻辑（账户变更时自动更新 sessionStorage）

#### `src/store/slices/deviceSlice.ts`（新建）

状态：`openDeviceIds`, `activeDeviceId`, `deviceStatuses`, `_pollingIntervalId`
操作：`openDeviceTab`, `closeDeviceTab`, `setActiveDeviceTab`, `addNewDevice`, `updateDevice`, `deleteDevice`, `pollDeviceStatuses`, `startStatusPolling`, `stopStatusPolling`

#### `src/store/slices/uiSlice.ts`（新建）

状态：`settings`, `isSettingsModalOpen`, `isDeviceModalOpen`, `deviceToEdit`, `isApiKeyModalOpen`
操作：所有 modal 开关、`updateSettings`, `updateBackendAISettings`, `addAgentUrlToHistory`

- 导出 `DEFAULT_SETTINGS` 常量，供 `useStore.ts` 的 persist merge 使用

#### `src/store/slices/sessionSlice.ts`（新建，含 ProgressToast 修复）

状态：`rollbackTarget`, `isWriteStartupModalOpen`, `writeStartupDeviceId`
操作：`promptRollback`, `cancelRollback`, `executeRollback`, `saveSessionAndAudit`, `openWriteStartupModal`, `closeWriteStartupModal`, `executeWriteStartup`

**ProgressToast 修复（详见第四节）**在此 slice 中完成。

#### `src/store/slices/scriptSlice.ts`（新建）

无状态（数据存储在 DataSlice），仅操作：`createScript`, `updateScript`, `deleteScript`

#### `src/store/slices/taskSlice.ts`（新建）

无状态（数据存储在 DataSlice），仅操作：`createScheduledTask`, `updateScheduledTask`, `deleteScheduledTask`

### `src/store/useStore.ts`（重写，627 行缩减为 33 行）

```typescript
export const useStore = create<FullStore>()(
  persist(
    (set, get, api) => ({
      ...createAuthSlice(set, get, api),
      ...createDataSlice(set, get, api),
      ...createDeviceSlice(set, get, api),
      ...createUiSlice(set, get, api),
      ...createSessionSlice(set, get, api),
      ...createScriptSlice(set, get, api),
      ...createTaskSlice(set, get, api),
    }),
    {
      name: 'chaintrace-settings-storage',
      storage: createJSONStorage(() => localStorage),
      partialize: (state) => ({ settings: state.settings }),
      merge: (persisted, current) => ({
        ...current,
        settings: { ...DEFAULT_SETTINGS, ...(persisted as Partial<FullStore>).settings },
      }),
    },
  ),
);
```

新增了 `merge` 函数：确保 localStorage 中的 settings 部分与 `DEFAULT_SETTINGS` 深度合并，避免新增默认 settings 字段在旧用户首次升级时丢失。

### 防循环导入设计

```
storeTypes.ts    ← 被所有 slice 导入（仅类型，无运行时依赖）
    ↑ 导入类型
slices/*.ts      ← 实现各自的 StateCreator
    ↑ 被聚合
useStore.ts      ← 最终组合所有 slice
```

`storeTypes.ts` 不导入任何 slice 文件，故无循环依赖。

## 四、ProgressToast 虚假进度修复（P2）

### 问题描述

原 `saveSessionAndAudit` 代码中（位于 `useStore.ts`，现移至 `sessionSlice.ts`），在发出 HTTP 请求**之前**就展示了"步骤一完成"的中间状态：

```typescript
// 问题代码（已删除）
const updatedSteps = [
  { text: '从设备获取最新配置', status: 'done' },   // ← 此时请求还未发出！
  { text: 'AI 智能审计分析', status: 'loading' },
  { text: '写入区块链', status: 'pending' },
];
toast.custom(...updatedSteps...);  // 立即展示，欺骗用户
```

这会让用户以为"从设备获取配置"已经完成，实际上后端还没开始处理。

### 修复方案

删除上述两行中间状态更新，保持 `initialSteps`（步骤一 loading）直到服务器响应返回：

```typescript
// 修复后：仅保留初始状态，直到响应成功
const initialSteps = [
  { text: '从设备获取最新配置', status: 'loading' },  // 保持 loading
  { text: 'AI 智能审计分析', status: 'pending' },
  { text: '写入区块链', status: 'pending' },
];
toast.custom(...initialSteps...);

// 发出请求...
const response = await apiFetch(...);
// 响应成功后，直接展示全部完成的 finalSteps
```

## 五、测试顺序文档说明（P3）

### 问题描述

`test_new_features.py` 中的 `TestScriptCRUD` 和 `TestScheduledTaskCRUD` 类的测试方法具有严格的执行顺序依赖（先创建再更新再删除），但缺少说明，容易让后续维护者误以为可以单独运行某个测试。

### 修复方案

为两个测试类的 docstring 添加详细说明：

```python
class TestScriptCRUD:
    """T10-A - 验证 /api/scripts 的完整 CRUD 操作。

    IMPORTANT — these tests are intentionally sequential and share state via
    class-level SCRIPT_ID.  pytest runs methods in definition order within a
    class, which is the only correct execution order here:
      1. list (empty)  2. create  3. duplicate → 409  4. list (populated)
      5. update        6. update nonexistent → 404   7. operator → 403
      8. delete        9. list (empty again)

    Each test inherits the DB state left by the previous one.  Re-ordering or
    running individual tests in isolation will produce unexpected failures.
    """
```

`TestScheduledTaskCRUD` 同理。
