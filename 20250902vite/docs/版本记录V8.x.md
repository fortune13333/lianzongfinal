# “链踪” V8.x 版本更新记录

## V8.0: 针对性合规策略 (Targeted Compliance Policies)

本次更新是“链踪”项目在企业级治理能力上的一次重大飞跃。我们采纳了“设备中心模型”的设计哲学，对平台进行了全面的架构升级，将原有的“一刀切”全局合规策略，演进为可针对不同设备、不同场景应用的**“靶向策略”**。

这使得“链踪”能够实现更精细化、更灵活、更贴近真实运维场景的合规治理。

### 核心变更详情

#### 1. 后端架构升级 (Backend & Database)

-   **数据库模型演进**:
    -   在数据库层面，我们通过引入一个新的 `device_policy_association` 关联表，建立了 `devices` (设备) 与 `policies` (策略) 之间的**多对多关系**。
    -   现在，一台设备可以关联多条策略，一条策略也可以被多台设备所应用，完美地映射了真实世界的复杂关系。

-   **核心审计逻辑重构**:
    -   合规性审计引擎（位于 `services.txt`）现在变得更加智能。在对配置进行审计时，它不再获取系统中的**所有**策略，而是**仅查询并应用与当前被操作设备明确关联的策略集**。
    -   这确保了审计的精确性，避免了不相关策略的干扰，并提升了性能。

-   **API 增强**:
    -   设备管理相关的API端点 (`/api/devices`) 进行了升级。现在，创建和更新设备时，请求体中可以包含一个 `policyIds` 数组，用于指定该设备需要绑定的策略。
    -   所有返回设备信息的API响应中，现在也都会包含该设备已关联的 `policyIds` 列表，确保了前后端数据的同步。

#### 2. 前端体验升级 (Frontend & UI/UX)

-   **设备管理界面重构**:
    -   “添加新设备”与“编辑设备”的弹窗 (`DeviceEditModal.tsx`) 进行了彻底重构，新增了 **“合规策略”** 配置区。
    -   管理员现在可以通过一个直观的多选框列表，轻松地为每一台设备**勾选或取消**其需要遵守的合规策略。

-   **提升信息透明度**:
    -   在设备详情页 (`DeviceDetails.tsx`)，我们增加了一个新的信息模块，用于清晰地展示**当前设备正在应用的所有合规策略**。
    -   这为网络工程师提供了一个完整的“设备合规档案”，让他们在操作设备时，能够对该设备所需遵守的规则一目了然。

## V8.1: 特权操作员与原子化权限 (Privileged Operator & Atomic Permissions)

本次迭代是对“链踪”权限管理体系的一次**根本性重构**。我们采纳了 V8.5 前瞻中的“特权操作员”务实方案，在不引入自定义角色的复杂性的前提下，实现了对权限的**原子化下放**，完美遵循了“最小权限原则”。

### 核心变更详情

#### 1. 后端架构升级：引入“额外权限”层

-   **数据库模型微调**:
    -   在 `users` 表中，增加了一个新的 `extra_permissions` 字段。
    -   该字段以逗号分隔的字符串形式，为 `operator` 角色的用户存储其被授予的、原本只有 `admin` 才拥有的特定权限（如 `device:delete`, `user:manage`）。

-   **授权逻辑重写**:
    -   我们创建了一个全新的、更智能的API授权依赖 `require_permission(permission: str)`。
    -   该依赖的检查逻辑为：**首先**，检查用户是否为 `admin`，若是则立即放行；**其次**，若用户为 `operator`，则检查其 `extra_permissions` 字段是否包含当前API所需的原子权限。
    -   所有受保护的API端点均已升级，使用此新依赖进行精细化的权限检查。

#### 2. 前端体验革命：从“角色”到“能力”的动态UI

-   **UI不再依赖硬编码角色**:
    -   我们彻底根除了前端代码中所有 `currentUser.role === 'admin'` 这样的硬编码逻辑。
    -   引入了全新的权限检查工具函数 `hasPermission()` 和 `canViewAdminPanel()` (位于 `src/utils/permissions.ts`)。现在，UI元素的显示/隐藏（如“删除设备”按钮、“管理中心”入口等）完全取决于当前用户是否拥有执行该操作的**具体权限**，而非其笼统的**角色**。

-   **管理员的“权力法杖”**:
    -   “用户编辑”弹窗 (`UserEditModal.tsx`) 进行了重大升级。当管理员编辑一个 `operator` 用户时，会看到一个全新的“额外权限”区域，其中包含了所有可供下放的原子权限复选框。
    -   这使得管理员可以像“赋予能力”一样，为特定的操作员精准地开启或关闭某项高级功能。

-   **智能的管理中心**:
    -   “管理中心” (`AdminPanel.tsx`) 现在能智能地感知当前用户的权限。例如，一个只被授予了 `user:manage` 权限的操作员，进入管理中心后将只能看到“用户管理”这一个选项卡。

### 状态与影响

经过 V8.1 的升级，“链踪”的权限系统变得极为灵活且安全。它允许组织根据实际需求，安全地将特定的管理职责下放给受信任的操作员，而无需授予其全部的管理员权限，这是项目迈向成熟企业级治理平台的关键一步。

### V8.1.1: 权限状态实时同步修复

-   **问题描述**: 修复了一个严重权限漏洞。当管理员在后台移除了某个已登录操作员的“额外权限”后，该操作员的界面并不会实时更新，导致其在会话有效期内，依然能使用已被撤销的权限。
-   **根源分析**: 问题的根源在于前端的状态管理。当一个操作员登录后，其用户信息（包括权限）被加载到 `currentUser` 状态中。但后台轮询只更新了 `allUsers` 列表，却忘记了去同步更新那个已经登录的操作员自己的 `currentUser` 对象。
-   **解决方案**: 我们对前端的中央数据存储逻辑 (`src/store/useStore.ts`) 进行了精准升级。现在，每当应用从后端轮询获取最新数据时，它不仅会更新全局的用户列表，还会**智能地找到当前登录用户对应的最新信息，并用它来刷新 `currentUser` 状态**，确保了用户的权限状态始终与后端数据库保持严格同步。

---

### V8.1.2: 无限循环加载与性能修复

-   **问题描述**: 修复了在 V8.1.1 的权限同步修复中，意外引入的一个性能问题。该问题导致前端在获取数据后，会陷入一个“无限循环加载”的状态，不断向后端发送 `GET /api/data` 请求，导致UI卡死。
-   **根源分析**: `useStore.ts` 中的 `fetchData` 函数在每次获取数据后，都会**无条件地**创建一个**新的** `currentUser` 对象来更新状态。即使新旧用户信息完全一样，但因为它们是两个不同的对象引用，React 依然判定为状态发生了变化，从而触发了 `App.tsx` 中依赖于 `currentUser` 的 `useEffect`，导致了重新获取数据的死循环。
-   **解决方案**: 我们对 `fetchData` 函数进行了优化。现在，在每次获取数据后，它会**先通过 `JSON.stringify` 比较新获取的用户信息和当前存储的是否真的有变化**。只有在用户信息（例如权限）确实被管理员修改时，它才会去更新 `currentUser` 状态，从而彻底打破了无限循环，恢复了应用的正常性能。

### V8.2: “断连自动审计”与审计闭环 (Auto-Audit on Disconnect)

本次更新引入了一项关键的安全增强功能，旨在彻底堵上因用户意外断开连接或恶意关闭浏览器而可能导致的审计漏洞，确保每一次交互式会话的最终状态都能被忠实记录。

#### 核心变更详情

1.  **引入“脏”会话跟踪机制**:
    -   后端现在会智能地跟踪每一个交互式WebSocket会话。只有当用户输入了可能修改设备配置的命令（如`configure terminal`而非`show`命令）后，该会话才会被标记为“脏”状态。

2.  **实现“断连自动审计”**:
    -   当一个被标记为“脏”的会话因任何原因（正常关闭、浏览器崩溃、网络中断）断开时，后端会**自动触发**一次最终的审计流程。
    -   该流程会立即重新连接设备，获取其当前的`running-config`，并将其作为一个**特殊类型的区块**记录到审计链中。

3.  **全新的区块类型与UI展示**:
    -   自动生成的区块其`changeType`会被标记为`auto_audit`。
    -   在前端UI的“配置历史”列表中，这类区块会带有一个独特的**“自动快照”**图标和标签，以明确区分其来源。
    -   为防止将一个可能不完整的、因意外中断而产生的配置作为恢复点，所有“自动快照”区块的**“回滚至此版本”功能均被禁用**。

4.  **管理员可控的AI分析模式**:
    -   “断连自动审计”的核心目标是**确保记录**，而非执行严格的治理。因此，该流程会**智能地跳过**可能导致记录失败的“合规性检查”。
    -   同时，为了平衡信息丰富度与资源消耗，我们为管理员在“设置”中心新增了一个选项，可以为该功能配置AI分析模式：
        -   **“尽力而为” (默认)**: 尝试执行AI智能分析，即使失败也继续保存区块。
        -   **“完全禁用”**: 始终跳过AI分析，只记录原始配置快照。

#### 状态与影响

通过引入“断连自动审计”机制，“链踪”实现了**审计的闭环**。它不仅作为一个强大的安全网，确保了恶意操作无法通过强制退出逃避追溯，也作为一个友好的“自动保存”功能，为因意外情况中断工作的工程师保留了最后的操作现场，极大地增强了平台的健壮性、安全性和用户体验。

### V8.3: 增强的实时命令拦截引擎 (Enhanced Real-time Command Interception Engine)

本次更新是对“链踪”实时防御能力的一次根本性重构。基于用户反馈，原有的、基于“启发式猜测”的“实时巡警”功能被证明是无效且不可靠的。本次迭代将其彻底移除，并替换为一个**功能强大、性能卓越、且完全由管理员配置驱动的全新实时命令拦截引擎**，为交互式终端构建了坚实的第一道防线。

#### 核心变更详情

1.  **引入 `config.ini` 驱动的规则库**:
    -   **从“猜测”到“确定”**: 我们彻底废除了旧的、尝试从自然语言策略中提取关键字的不可靠机制。取而代之的是，在 `config.ini` 文件中新增了一个专属的 `[interception_rules]` 配置区。
    -   **管理员全权掌控**: 这种“配置即代码”的设计，将定义实时拦截规则的权力完全交还给管理员，确保了拦截逻辑的**100%确定性与可靠性**。

2.  **丰富的拦截规则类型**:
    -   为了提供最大的灵活性以应对各种复杂的拦截场景，新的引擎支持三种强大的匹配类型：
        -   **`contains`**: 只要命令中**包含**指定的任意一个关键字，就拦截。
        -   **`startswith`**: 只要命令**以**指定的任意一个关键字开头，就拦截（非常适合拦截`erase`, `reload`等一整类命令）。
        -   **`regex`**: 允许管理员使用**正则表达式**来匹配和拦截最复杂的命令模式。

3.  **详尽的内置文档**:
    -   为了降低使用门槛，我们在 `config.ini` 文件的 `[interception_rules]` 配置区顶部，直接嵌入了详尽的、中英双语的使用说明和丰富的示例。

4.  **后端集成与性能**:
    -   后端服务 (`core.txt`) 在启动时会一次性地加载并解析所有规则（包括编译正则表达式），将其存入高效的内存结构中。
    -   WebSocket 处理器 (`websocket_handler.txt`) 的核心逻辑被重构，现在会对用户输入的每一条命令进行快速的内存匹配检查。该检查过程在纳秒级完成，对实时交互体验**无任何性能影响**。

#### 状态与影响

经过 V8.3 的升级，“链踪”的实时防御能力发生了质的飞跃。它将一个不可靠的功能，转变为平台“事前治理”理念的基石。现在，任何不合规或危险的命令，都会在按下回车的一瞬间被精准、可靠地拦截，彻底杜绝了其进入设备`running-config`的可能性，极大地增强了平台的整体安全性。
