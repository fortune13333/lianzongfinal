# “链踪”V8.5 前瞻：权限原子化下放的务实实现方案 (可实现)

本文档记录了在构想了宏大的V9.0权限系统蓝图后，开发者与AI助手之间关于寻找一个更轻量、更务实、投入产出比更高的中间方案的深度技术评估。这次对话为项目的下一个迭代版本（V8.5）规划了一个复杂度可控且价值巨大的“特权操作员”权限模型。

让我们把这个方案称为**V8.5 的“特权操作员 (Privileged Operator)”模型**。

---

### **评估结论：完全可行，复杂度可控，是一次“改良性”而非“革命性”的升级**

这个方案的核心思想是：**我们不改变现有的 `admin` / `operator` 双角色体系，而是在其之上，为 `operator` 增加一个“权限覆盖 (Permission Override)”层。**

这意味着，一个用户的基础角色仍然是 `operator`，但管理员可以像授予“荣誉勋章”一样，将一两个特定的、原本只有 `admin` 才拥有的权限，单独授予这位操作员。

---

### **V8.5 蓝图：一个务实的“特权操作员”实现方案**

这个方案的美妙之处在于，它对现有架构的改动极小。

#### **第一步：数据库模型的“微创手术”**

我们**不需要**像 V9.0 方案那样引入三个全新的表。我们只需要对现有的 `models.txt` 中的 `users` 表，做一个极小的改动：

*   **在 `User` 表中增加一个字段**:
    *   **字段名**: `extra_permissions`
    *   **类型**: `Text` 或 `String`
    *   **作用**: 这个字段将以**逗号分隔的字符串**形式，存储授予该用户的额外权限。例如：`"device:delete,user:manage"`。

**影响**: 这次数据库变更极其简单，只需要一次“添加列”的迁移操作，完全不会影响现有数据。

#### **第二步：后端授权逻辑的“精准升级”**

我们的授权逻辑需要变得更聪明一些，但无需完全重写。

1.  **定义“原子权限”**:
    我们首先要在后端代码中，明确地定义出有哪些“原子权限”是可以被下放的。这个列表是固定的，例如：
    *   `device:create`
    *   `device:update`
    *   `device:delete`
    *   `user:manage` (涵盖增删改查)
    *   `template:manage`
    *   `policy:manage`
    *   `rollback:execute`
    *   `system:reset`

2.  **创建一个新的授权依赖**:
    我们可以创建一个新的、更智能的依赖，例如 `require_permission(permission: str)`。当一个API请求进来时，它的检查逻辑将是：

    *   **第一步：检查是否为`admin`**: 获取当前用户。如果他的 `role` 是 `admin`，**则立即放行**。这确保了`admin`的超级权限保持不变。
    *   **第二步：如果不是`admin`，则检查“额外权限”**:
        *   从数据库中读取该用户的 `extra_permissions` 字段。
        *   将这个字符串（如 `"device:delete,user:manage"`）按逗号分割成一个列表。
        *   检查当前API所要求的权限（例如 `device:delete`）是否在这个列表中。
        *   **如果是，则放行；如果不是，则拒绝。**

3.  **适配 API 端点 (`api_routes.txt`)**:
    这是本次升级的主要工作。我们需要审查所有受保护的API，将原有的 `Depends(verify_admin_dep)` 替换为新的依赖。例如：
    *   `DELETE /api/devices/{device_id}` 接口的依赖，将从 `verify_admin_dep` 变为 `require_permission("device:delete")`。
    *   `POST /api/users` 接口的依赖，将变为 `require_permission("user:manage")`。

#### **第三步：前端UI的“轻量级改造”**

前端的改动也同样精准而微小。

*   **改造“编辑用户”弹窗 (`UserEditModal.tsx`)**:
    *   当管理员编辑一个**角色为 `operator`** 的用户时，弹窗中会出现一个新的区域，标题为“授予的额外权限 (Granted Extra Permissions)”。
    *   这个区域会以**复选框 (Checkbox)** 的形式，列出所有可供下放的原子权限（就是我们在后端定义好的那个列表）。
    *   管理员可以为这位`operator`勾选他需要的特定权限。
    *   **这个区域在编辑`admin`用户，或者`operator`自己编辑自己时，是不可见或不可用的。**
*   **更新API负载**:
    *   当管理员保存更改时，前端会将所有勾选的权限，组合成一个逗号分隔的字符串，通过 `PUT /api/users/{user_id}` 接口发送给后端。
