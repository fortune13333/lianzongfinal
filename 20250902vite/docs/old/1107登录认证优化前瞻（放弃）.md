一、 当前登录方式的“三大安全隐患”
在设计优化方案之前，我们必须清醒地认识到当前登录流程中存在的三个“致命”安全问题。
1. 密码明文存储 (Cleartext Password Storage)
现状: 在后端的数据库 (chaintrace.db) 的 users 表中，用户的密码是以明文形式直接存储的。
风险: 这是安全设计中最不可饶恕的“原罪”。一旦数据库文件被任何未经授权的人员获取（无论是外部攻击还是内部泄露），所有用户的账户和密码将瞬间暴露无遗。
好比是: 您把所有保险柜的密码，都用记号笔直接写在了保险柜的门上。
2. 密码明文传输 (Cleartext Password Transmission)
现状: 当您在登录页面输入密码并点击登录时，您的密码在网络中是以明文形式，通过HTTP请求从浏览器发送到后端代理的。
风险: 在任何一个中间网络节点（如不安全的Wi-Fi、被渗透的路由器），攻击者都可以通过抓包工具，像看明信片一样，直接读到您的用户名和密码。
好比是: 您把银行卡密码写在一张明信片上，然后邮寄给银行。
3. 脆弱的会话管理 (Weak Session Management)
现状: 用户登录成功后，前端只是在浏览器的 sessionStorage 中存入了一个包含用户信息的对象。后续的所有操作，都是基于这个简单的前端“标记”。
风险: 这种方式非常脆弱。它没有过期时间，无法被后端主动吊销，并且容易受到跨站脚本（XSS）等攻击的窃取和伪造。
好比是: 保安在您进入大楼时，只给了您一张手写的、没有照片、没有有效期的纸条作为通行证。
二、 三步走的、行业标准的优化方案
为了彻底解决上述问题，我们需要对整个认证与授权体系进行一次全面的、符合行业最佳实践的“安全加固”升级。
方案一：密码的“单向哈希”存储 (最核心、最优先)
这是解决“明文存储”问题的唯一正确方法。
核心原理: 我们不再存储密码本身，而是存储密码经过加盐哈希 (Salted Hashing) 后的一个“指纹”字符串。哈希是一种单向加密算法，无法从“指纹”反推出原始密码。
实施思路:
后端改造 (crud.txt):
在创建新用户或修改用户密码时，后端不再直接保存用户提交的密码。
而是使用一个标准的密码处理库（如Python的passlib），为该密码生成一个唯一的“盐”，然后进行哈希计算，最终将生成的、包含了盐和哈希值的长字符串存入数据库。
登录逻辑改造 (api_routes.txt):
当用户尝试登录时，后端接收到用户提交的明文密码。
后端从数据库中根据用户名，找到存储的那个“哈希字符串”。
passlib库会从这个字符串中自动提取出“盐”，然后用同样的方法处理用户刚刚提交的密码，计算出一个新的哈希值。
最终，后端只比较两个哈希值是否相等。如果相等，则证明密码正确。
效果: 即使数据库被完全泄露，攻击者得到的也只是一堆无法破解的哈希字符串，用户的真实密码依然是安全的。
方案二：传输通道的“全程加密” (HTTPS)
这是解决“明文传输”问题的标准方案。
核心原理: 使用HTTPS（HTTP over SSL/TLS）协议。它会在您的浏览器和服务器之间，建立一个端到端的加密通道。
实施思路:
这不是代码层面的修改，而是部署层面的配置。
我们需要在后端代理服务器的前面，部署一个反向代理服务器（如Nginx）。
为您的服务申请并配置一个SSL证书（可以使用Let's Encrypt免费获取）。
Nginx将负责处理所有的HTTPS加密和解密，然后将解密后的、安全的HTTP请求转发给我们的FastAPI后端。
效果: 任何中间人都无法再窃听浏览器和服务器之间的通信内容，包括密码。
方案三：会话管理的“无状态令牌” (JWT)
这是解决“脆弱会话”问题，并实现现代化、可扩展API认证的最佳实践。
核心原理: 使用JSON Web Tokens (JWT)。JWT是一个紧凑、独立、安全的字符串，它本身就包含了用户的身份信息、权限和一个过期时间，并且带有一个无法伪造的数字签名。
实施思路:
新的登录流程:
用户提交用户名和密码。
后端验证密码正确后（通过哈希比对），不再只是返回“成功”。
而是生成一个JWT，这个JWT中包含了user_id、role、extra_permissions以及一个过期时间戳（例如，8小时后过期）。然后，用一个只有服务器知道的秘密密钥 (Secret Key) 对这个JWT进行签名。
后端将这个签了名的JWT字符串，返回给前端。
前端存储与发送:
前端收到JWT后，将其存储在localStorage中。
从此以后，前端的每一次API请求，都必须在HTTP请求头中附带上这个JWT，格式通常是 Authorization: Bearer <jwt_token>。
后端验证:
后端的每一个需要认证的API接口，都会先检查请求头中的Authorization。
它会用自己保存的那个秘密密钥，来验证JWT的签名是否合法、是否未被篡改，并检查其是否已过期。
只有在JWT完全有效的情况下，后端才会执行该API的业务逻辑。
效果:
无状态 (Stateless): 后端无需再记录任何会话信息，极大地提升了可扩展性。
安全: 签名机制防止了令牌被伪造。
自过期: 令牌自带过期时间，自动处理了会话超时问题。
