#### **5.1 测试环境搭建**

&nbsp;&nbsp;&nbsp;&nbsp;为确保测试的真实性和有效性，我搭建了如下测试环境：

*   **软件环境**:
    *   前端运行环境: Google Chrome 浏览器 (版本 120+), Node.js v20.x。
    *   后端运行环境: Windows 11 专业版, Python 3.10。
    *   依赖库: 我严格按照`requirements.txt`和`package.json`文件中定义的版本安装了所有依赖库。
*   **硬件环境**:
    *   客户端与服务器（同一台物理机）: Intel Core i7-12700H CPU @ 2.30GHz, 32GB RAM。
*   **模拟网络环境**:
    *   为模拟真实的多厂商网络环境，我使用了**GNS3 (Graphical Network Simulator-3)**网络模拟软件。
    *   在GNS3中，我部署了多个虚拟化的网络设备镜像，主要包括 **Cisco IOSv** (用于模拟路由器/交换机) 和 **H3C Comware V7**，以验证系统的多厂商支持能力。
    *   我将所有虚拟设备通过GNS3的“云”节点桥接到本地物理网络的`Microsoft KM-TEST Loopback Adapter`，并为它们配置了与后端代理服务器在同一网段的IP地址，确保后端代理能够通过SSH协议顺利访问它们。

#### **5.2 功能测试**

&nbsp;&nbsp;&nbsp;&nbsp;我针对系统的四大核心功能模块，设计并执行了以下测试用例。

##### **5.2.1 核心审计链功能测试**

*   **测试用例1：新区块创建与链接**
    *   **操作**: 我对一台Cisco IOSv设备进行多次配置修改（如修改`hostname`、添加`loopback`接口）并执行“保存并审计”。
    *   **预期结果**: 每一次操作都在数据库的`blocks`表中生成一条新的记录。通过DB Browser for SQLite工具检查，新区块的`index`正确递增，其`prev_hash`字段与前一区块的`hash`字段完全匹配。
    *   **测试结果**: **通过**。测试结果符合预期。

*   **测试用例2：链完整性验证（正常情况）**
    *   **操作**: 我对一条未经修改的区块链执行“验证链完整性”功能。
    *   **预期结果**: 验证模态框显示所有区块均验证通过，最终提示“验证成功！区块链完整且未被篡改。”
    *   **测试结果**: **通过**。测试结果符合预期。

*   **测试用例3：链完整性验证（数据篡改）**
    *   **操作**: 我使用DB Browser for SQLite工具，直接修改`blocks`表中某历史区块的`data`字段，将其中一个接口的IP地址修改了一个数字。
    *   **预期结果**: 执行验证时，被篡改的区块显示验证失败，并明确指出“存储值”与“计算值”不匹配。最终提示“验证失败！发现被篡改的区块。”
    *   **测试结果**: **通过**。测试结果符合预期，证明了我实现的哈希验证机制的有效性。

*   **测试用例4：智能回滚**
    *   **操作**: 我以管理员身份，将设备回滚至一个较早的历史版本。
    *   **预期结果**: 系统成功创建一个`changeType`为`rollback`的新区块，其配置内容与目标版本一致。打开该区块详情，AI分析部分包含结构化的“回滚计划”。
    *   **测试结果**: **通过**。测试结果符合预期。

##### **5.2.2 AI治理功能测试（合规性拦截）**

*   **测试用例1：策略定义与靶向应用**
    *   **操作**: 我创建了一条名为“禁止使用Telnet”的策略，并仅将其关联到设备`CISCO-01`。
    *   **预期结果**: 策略成功创建。在`CISCO-01`的设备编辑页可以看到该策略已勾选，而另一台设备`H3C-01`则没有。
    *   **测试结果**: **通过**。测试结果符合预期。

*   **测试用例2：违规变更拦截**
    *   **操作**: 我尝试向设备`CISCO-01`提交一段包含`transport input telnet`的配置。
    *   **预期结果**: 提交失败，前端弹出错误提示，明确指出违反了“禁止使用Telnet”策略。数据库中未生成新区块。
    *   **测试结果**: **通过**。测试结果符合预期。

*   **测试用例3：合规变更通过**
    *   **操作**: 我尝试向设备`CISCO-01`提交一段不含`telnet`的合规配置。
    *   **预期结果**: 提交成功。新区块的详情中包含一份“合规性审计报告”，显示所有关联策略均已通过。
    *   **测试结果**: **通过**。测试结果符合预期。

*   **测试用例4：回滚操作绕过审计**
    *   **操作**: 假设版本5引入了“禁止Telnet”策略。我尝试将设备回滚至一个包含Telnet配置的版本4。
    *   **预期结果**: 回滚操作**成功**，系统创建了新的回滚区块，并未因违反“禁止使用Telnet”策略而被拦截。
    *   **测试结果**: **通过**。测试结果符合预期，验证了我设计的“应急恢复优先”原则。

#### **5.3 性能、安全性与健壮性分析**

*   **性能分析**:
    *   **前端性能**: 应用加载速度快，UI交互流畅。我通过引入Zustand进行状态管理，并使用`useMemo`等钩子对高计算量组件（如并排差异对比）进行优化，有效避免了不必要的组件重渲染，保证了应用的响应速度。
    *   **后端性能**: FastAPI与Uvicorn的组合提供了高性能的API服务。对于本项目预设的企业内部中低并发使用场景，其性能完全满足需求。
    *   **AI响应时间**: 系统性能的主要延迟来源于与外部Gemini AI服务的网络通信。我的测试表明，一次标准的合规审计或智能分析的耗时通常在5-15秒之间，这在可接受范围内。我设计并实现的`ProgressToast`分步反馈机制，有效缓解了用户在等待过程中的焦虑感。具体测试数据如表5-1所示。

*   **安全性分析**:
    我从多个维度对系统的安全性进行了设计与加固。
    *   **访问控制**: 我实现的基于角色的精细化权限控制，确保了不同用户只能访问其被授权的资源和操作。
    *   **数据传输**: 前端与后端、后端与网络设备之间的通信均基于标准协议（HTTP/WebSocket, SSH），可在生产环境中通过部署TLS证书和配置SSH密钥对来进一步加密和加固。
    *   **凭证管理与代码安全**: 我将敏感的设备登录凭证存储在与代码分离的`config.ini`文件中，避免了硬编码风险。此外，后端数据库操作全部通过SQLAlchemy ORM进行，其参数化查询机制从根本上杜绝了SQL注入的风险。
    *   **操作安全**: 我设计的高危命令拦截和“事前治理”引擎，有效防止了因误操作或违规操作导致的安全事故。
    *   **审计与追溯**: 我实现的不可变审计链和详尽的管理行为日志，为事后追溯和安全审计提供了强有力的、可信的技术保障。

*   **健壮性分析 (Robustness Analysis)**:
    我在设计中充分考虑了异常情况下的稳定运行能力，并通过测试验证了容错机制的有效性。
    *   **AI服务调用弹性**: 我通过工具模拟了网络不佳和AI服务返回临时性错误（如503 Service Unavailable）的场景。测试结果表明，我实现的“指数退避重试”机制被成功触发，系统能够承受短时间的AI服务中断，并在服务恢复后自动完成之前失败的请求，表现出良好的容错能力。
    *   **数据库架构自适应检查**: 我通过手动修改数据库文件（例如，删除一个数据表列）来模拟版本不匹配的场景。测试结果表明，我设计的后端服务在启动时能够成功检测到架构不匹配，并立即停止运行，同时在控制台输出清晰的中文错误指引。此机制有效防止了因数据模型不兼容导致的运行时严重错误。
