什么是自动化测试
简单说：写代码来测试你的代码，而不是每次手动打开浏览器点点点。运行一个命令，它自动验证所有功能是否正常，并输出哪些通过、哪些失败。

测试分层
对于这个项目，测试分三层，工作量和价值不同：

第一层：单元测试（Unit Test）
测试单个函数，不依赖数据库或网络。

适合测试的目标：

core.py 中的 verify_password()、hash_password() — 输入明文，验证哈希结果是否能通过校验
core.py 中的 is_simulation_mode() — 给定不同的 config.ini 内容，验证返回值是否正确
crud.py 中 tags 的逗号拼接/拆分逻辑 — 输入 ["prod", "core"]，验证存储为 "prod,core"，反向解析回 ["prod", "core"]
services.py 中区块链完整性验证逻辑 — 构造一个哈希链，验证 verify 函数能识别出被篡改的区块
第二层：集成测试（Integration Test）
启动真实的数据库（用内存 SQLite），测试 API 端点。

适合测试的目标：

POST /api/login — 正确密码返回 200 + token，错误密码返回 401
POST /api/devices — 无权限用户返回 403，有权限用户创建成功返回 201
GET /api/devices/poll-status — 返回格式符合预期
PUT /api/devices/{id} — 更新 tags 后再 GET，标签确实被保存
添加区块 → 验证完整性 → 篡改一个区块 → 验证完整性应报错
工具： FastAPI 自带 TestClient（基于 httpx），配合 pytest 使用，不需要启动真实服务器。

第三层：端到端测试（E2E Test）
模拟真实浏览器操作，通常用 Playwright 或 Cypress。

对毕业设计来说性价比低，不建议做。

建议方案（性价比最高）
只做前两层，重点放在集成测试，覆盖以下 8 个测试用例：

编号	类型	测试内容
T01	单元	密码哈希与验证
T02	单元	tags 字符串与列表互转
T03	单元	区块链完整性验证（正常链）
T04	单元	区块链完整性验证（被篡改链）
T05	集成	登录接口：正确/错误密码
T06	集成	设备增删改查（需 admin token）
T07	集成	无 token 访问受保护接口返回 401
T08	集成	operator 角色无法访问 admin 接口返回 403
文件结构

agentv2/
├── tests/
│   ├── __init__.py
│   ├── test_unit.py       # T01-T04
│   └── test_api.py        # T05-T08
运行方式：


cd agentv2
pytest tests/ -v
对答辩的价值
8 个测试用例不多，但够用。论文里可以写：

设计了单元测试覆盖核心算法（区块链验证、密码哈希）
设计了集成测试覆盖 API 鉴权和权限控制
