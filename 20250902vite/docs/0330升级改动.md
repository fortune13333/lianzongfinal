## P0 — 安全性

### 1. CORS 限制：从通配符 `*` 改为可配置

**问题：** `agent.py` 原始配置 `allow_origins=["*"]`，任何来源都可跨域请求 API，生产环境存在安全风险。

**修改文件：**
- `agentv2/agent.py`
- `agentv2/config.ini`

**改动说明：**

`agent.py` 在创建 `CORSMiddleware` 之前，先从 `config.ini [security]` 读取 `allowed_origins` 配置项：

```python
_cors_origins_str: str = config.get('security', 'allowed_origins', fallback='').strip()
if _cors_origins_str:
    _cors_origins = [o.strip() for o in _cors_origins_str.split(',') if o.strip()]
else:
    _cors_origins = ["*"]
    logging.warning("安全警告: [security] allowed_origins 未配置，CORS 已设为允许所有来源 (*)...")
```

`config.ini` 新增说明注释和配置项（默认注释状态，开箱为开发模式）：

```ini
# 示例: allowed_origins = http://localhost:5173, http://192.168.1.100:8001
# allowed_origins = http://localhost:5173, http://127.0.0.1:8001
```

**使用方法：** 生产部署时在 `config.ini [security]` 中取消注释并填入实际域名/IP。

### 2. JWT Secret 支持环境变量

**问题：** JWT 密钥只能通过 `config.ini` 配置（明文文件），不符合生产安全最佳实践（密钥不应进入代码仓库）。

**修改文件：** `agentv2/core.py`

**改动说明：** 密钥加载优先级改为：**环境变量 > config.ini > 自动生成随机密钥**

```python
_env_jwt_key: str = _os.environ.get('CT_JWT_SECRET_KEY', '').strip()
_raw_jwt_key: str = config.get('security', 'jwt_secret_key', fallback='').strip()

if _env_jwt_key:
    JWT_SECRET_KEY = _env_jwt_key
    logging.info("JWT 密钥已从环境变量 CT_JWT_SECRET_KEY 加载。")
elif _raw_jwt_key:
    JWT_SECRET_KEY = _raw_jwt_key
    logging.info("JWT 密钥已从 config.ini [security] 加载。")
else:
    JWT_SECRET_KEY = _secrets.token_hex(32)
    logging.warning("安全警告: JWT Secret 未配置，已自动生成随机密钥...")
```

**使用方法：**
- 开发：保持 `config.ini` 中的 `jwt_secret_key` 不变
- 生产：设置环境变量 `CT_JWT_SECRET_KEY=你的强随机密钥`（可用 `python -c "import secrets; print(secrets.token_hex(32))"` 生成）
- Docker 部署：在 `docker-compose.yml` 的 `environment` 中配置

### 3. HTTPS / nginx 反向代理支持

**问题：** Docker 部署时服务直接暴露 HTTP，无 TLS 加密。

**新增/修改文件：**
- `nginx.conf`（新建）
- `docker-compose.yml`（修改）

**`nginx.conf` 核心内容：**
- HTTP (80) → HTTPS (443) 自动重定向
- 仅允许 TLSv1.2 / TLSv1.3
- 添加 `Strict-Transport-Security`、`X-Frame-Options DENY`、`X-Content-Type-Options nosniff` 安全响应头
- WebSocket `Upgrade` 头正确转发（终端会话需要）
- `proxy_read_timeout 3600s`（支持长连接 WebSocket 终端）

**`docker-compose.yml` 主要变更：**
- `chaintrace` 服务端口改为内部访问（不再直接暴露 8001）
- 新增 `healthcheck`（每 30s 检查 `/api/health`）
- 新增 `environment.CT_JWT_SECRET_KEY` 环境变量注入
- 新增 `nginx` 服务，监听 80/443，依赖 `chaintrace` 健康状态后启动

**使用方法：**

```bash
# 1. 生成自签名证书（测试用）
mkdir ssl
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout ssl/key.pem -out ssl/cert.pem -subj "/CN=chaintrace"

# 2. 启动（含 nginx HTTPS）
docker compose --profile https up -d

# 3. 仅启动后端（不需要 HTTPS 时）
docker compose up -d
```

## P1 — 稳定性

### 4. 修复 `datetime.utcnow()` 弃用警告

**问题：** Python 3.12+ 中 `datetime.utcnow()` 已被标记为弃用，应使用时区感知的 `datetime.now(timezone.utc)`。

**修改文件：** `agentv2/core.py`、`agentv2/api_routes.py`

**core.py 改动（第 218 行）：**
```python
# 修改前
expire = _datetime.datetime.utcnow() + timedelta(minutes=JWT_EXPIRE_MINUTES)

# 修改后
expire = datetime.now(timezone.utc) + timedelta(minutes=JWT_EXPIRE_MINUTES)
```

同时将 `import datetime as _datetime` 改为更精确的 `from datetime import datetime, timedelta, timezone`。

**api_routes.py 改动：** `poll_all_device_statuses`、`export_device_history`、`full_system_backup` 三个函数中共 4 处 `datetime.utcnow()` 全部替换为 `datetime.now(timezone.utc).isoformat().replace('+00:00', 'Z')`。

### 5. Alembic 数据库迁移管理

**问题：** 原来的数据库迁移方式是在 `agent.py` 启动时执行裸 SQL `ALTER TABLE devices ADD COLUMN tags TEXT`，无版本管理，不可回滚，不可追溯。

**新增/修改文件：**
- `agentv2/alembic.ini`（新建）
- `agentv2/alembic/env.py`（新建）
- `agentv2/alembic/script.py.mako`（新建，迁移文件模板）
- `agentv2/alembic/versions/001_initial_schema.py`（新建，初始迁移）
- `agentv2/requirements.txt`（新增 `alembic` 依赖）
- `agentv2/agent.py`（修改启动流程）

**`alembic/env.py` 核心设计：**
- 自动从 `database.py` 读取 `SQLALCHEMY_DATABASE_URL`
- 导入 `models` 模块确保所有 ORM 模型注册到 `Base.metadata`
- 启用 `render_as_batch=True`（SQLite 的 ALTER TABLE 必须通过 batch 模式实现）

**`versions/001_initial_schema.py` 特点：**
- 幂等操作：用 `inspector.get_table_names()` 检查表是否已存在，已存在则跳过创建
- 包含 `devices.tags` 列的补充添加逻辑（兼容升级前的旧数据库）
- `downgrade()` 支持完整回滚

**`agent.py` 启动流程变更：**
```python
# 修改前：裸 ALTER TABLE
try:
    mig_db.execute(_text("ALTER TABLE devices ADD COLUMN tags TEXT"))
    ...

# 修改后：Alembic upgrade
try:
    from alembic.config import Config as AlembicConfig
    from alembic import command as alembic_command
    alembic_cfg = AlembicConfig(str(Path(__file__).parent / "alembic.ini"))
    alembic_command.upgrade(alembic_cfg, "head")
except Exception as alembic_err:
    logging.warning(f"Alembic 迁移失败，回退到 SQLAlchemy create_all: {alembic_err}")
    init_db()  # fallback
```

**CLI 使用（手动迁移）：**
```bash
cd agentv2
# 查看当前版本
alembic current
# 升级到最新
alembic upgrade head
# 回滚一个版本
alembic downgrade -1
# 生成新迁移（修改 models.py 后）
alembic revision --autogenerate -m "描述变更"
```

### 6. 结构化日志系统

**问题：** 原来只有一行 `logging.basicConfig(level=logging.INFO, ...)`，日志只输出到控制台，无文件持久化，无大小限制。

_log_dir = Path(__file__).parent / "logs"
_log_dir.mkdir(exist_ok=True)

# 控制台 Handler（始终开启）
_console_handler = logging.StreamHandler()

# 滚动文件 Handler（单文件最大 5MB，保留 5 个备份）
_file_handler = _log_handlers.RotatingFileHandler(
    _log_dir / "chaintrace.log",
    maxBytes=5 * 1024 * 1024,
    backupCount=5,
    encoding="utf-8"

_formatter = logging.Formatter(
    fmt="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    datefmt="%Y-%m-%dT%H:%M:%S"

# 抑制第三方库噪音日志

### 10. GitHub Actions CI/CD 流水线

**问题：** 没有自动化流水线，合并代码后无法自动验证测试和构建。

**新增文件：** `.github/workflows/ci.yml`

**触发条件：**
- `push` 到 `main` 或 `dev` 分支
- 向 `main` 发起 `pull_request`

**三个 Job：**

```
backend-tests ─┐
               ├──► docker-build
frontend ──────┘
```

| Job | 运行环境 | 步骤 |
|-----|---------|------|
| `backend-tests` | ubuntu-latest, Python 3.11 | pip install → pytest -v |
| `frontend` | ubuntu-latest, Node 20 | npm ci → vitest run → vite build |
| `docker-build` | ubuntu-latest | docker build（依赖前两个 job 通过） |

**特别说明：** CI 中通过环境变量注入测试专用 JWT Secret，避免随机密钥导致 Token 不稳定：
```yaml

  CT_JWT_SECRET_KEY: ci-test-secret-not-for-production
```
